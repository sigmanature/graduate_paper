## 5.2 测试方法与负载设计

### 5.2.1 工作负载

实验以顺序读写（sequential read/write）为核心负载，使用 fio 工具生成单线程顺序访问请求。选择顺序读写作为主要评价负载的原因在于：本文的核心优化路径——Large Folio 与 iomap 区间映射——在连续文件访问场景下收益最为显著，页缓存对象粒度的扩大和映射次数的减少直接体现在大块连续 I/O 的吞吐能力上。

每次测试使用三类文件：普通文件、稀疏文件（sparse file）和压缩文件（compressed file），以覆盖本文实现所涉及的三种主要文件类型。测试文件大小均设置为大于系统可用内存，确保冷读（cold read）场景下每次读取均需从存储设备实际获取数据，避免页缓存命中掩盖路径开销差异。

### 5.2.2 测试矩阵

实验围绕以下维度的组合展开：

- **块大小（block size）**：4KB、16KB、64KB、128KB、256KB、512KB、1MB，覆盖从小粒度到大粒度连续访问的完整范围。
- **读写模式**：顺序写（sequential write）、冷顺序读（cold sequential read）、热顺序读（warm sequential read）。冷读在每轮测试前清理页缓存，热读则在第一轮读完成后立即进行第二轮读。
- **队列深度（queue depth）**：1、2、4、8、16，用于观察不同并发度下路径的处理能力。

### 5.2.3 评价指标

实验主要采集以下指标：

- **吞吐量（bandwidth）**：以 MiB/s 为单位，直接反映文件系统顺序 I/O 路径的数据传输能力。
- **IOPS**：每秒 I/O 操作次数，辅助衡量小粒度请求下的路径开销。
- **CPU 利用率**：包括用户态（usr）和内核态（sys）占比，用于评估改造路径是否引入了额外 CPU 开销。

## 5.3 实验结果与分析

### 5.3.1 QEMU 虚拟机平台

图 5-1 至图 5-3 展示了 QEMU 平台上原生 F2FS 与改造后 F2FS 在普通文件、稀疏文件和压缩文件三类场景下的顺序 I/O 带宽对比。

在顺序写入场景中，改造后路径在全部分块大小下均表现出高于原生路径的带宽值，且随着块大小从 4KB 增大至 1MB，带宽提升幅度呈扩大趋势。这一趋势与本文的设计预期一致：块大小越大，Large Folio 的一次性覆盖优势越明显，页缓存对象数量的减少和映射查询次数的下降直接贡献了吞吐提升。稀疏文件和压缩文件的顺序写趋势与普通文件一致，说明本文对写路径的区间化改造并未因文件类型差异而退化。

在冷顺序读场景中，改造后路径同样获得带宽提升，但提升幅度略低于顺序写。这一差异的主要原因在于读路径的瓶颈不仅取决于映射组织方式，还受到预读（readahead）窗口、bio 合并策略以及底层存储设备读延迟的共同影响。然而，改造后路径并未出现性能倒退，说明新的 readahead 和 folio 完成逻辑没有成为读路径的瓶颈。

在热顺序读场景中，由于数据已经驻留在页缓存中，改造前后的带宽差异较小。这符合预期：热读场景下磁盘 I/O 不参与，瓶颈主要受内存拷贝速率和 CPU 缓存行为支配，页缓存对象粒度的影响相对间接。

### 5.3.2 树莓派 5 平台

图 5-4 至图 5-6 展示了树莓派 5 平台上相同负载的带宽对比。总体趋势与 QEMU 环境一致：改造后路径在顺序写和冷顺序读场景中获得吞吐提升，热顺序读场景差异较小。这一结果说明本文方案的效果不依赖于虚拟化环境中的特定缓存行为，在真实 ARM 处理器和外接 SSD 的组合下仍能保持同方向变化。

树莓派 5 平台上的带宽绝对值低于 QEMU 环境，主要原因在于 USB 3.0 接口的带宽限制和外接 SSD 自身的性能上限。但改造前后的相对提升比例与 QEMU 环境相近，进一步验证了本文方案的可移植性。

### 5.3.3 CPU 利用率分析

除带宽外，实验还采集了顺序读写过程中的 CPU 利用率。改造后路径在内核态（sys）的 CPU 占比略有下降，这与以下因素有关：Large Folio 减少了页缓存对象数量，从而减少了锁获取、映射查询和 LRU 链表操作的次数；iomap 的区间迭代方式替代了逐块 buffer_head 遍历，降低了状态维护的 CPU 开销。用户态（usr）CPU 占比在改造前后基本持平，因为应用层的数据拷贝行为未受影响。

## 5.4 实验结果讨论

综合以上结果，可以得出以下几点结论。

第一，本文引入的 Large Folio 与 iomap 适配方案在顺序 I/O 场景下能够有效提升 F2FS 的吞吐表现，且收益随块大小增大而扩大，与“减少小粒度对象、降低元数据开销”的设计目标一致。

第二，改造后路径在普通文件、稀疏文件和压缩文件三类场景下均未出现性能倒退，说明映射转换、状态统一和写回控制机制在覆盖不同文件类型时保持了较好的一致性。

第三，跨平台验证结果表明，本方案的收益不限于特定虚拟化环境，在真实硬件平台上仍可复现，具有一定的通用性。

第四，应当指出的是，本实验以顺序读写为主要负载，尚未覆盖随机读写、多线程并发和混合读写等更复杂的访问模式。在这些场景下，Large Folio 的分配成功率、内存碎片的影响以及 iomap 迭代边界对非连续映射的处理效率，仍有待进一步实验验证。

## 5.5 本章小结

本章在两个平台上对改造前后的 F2FS 进行了对比测试。实验覆盖了不同块大小、不同读写模式和不同文件类型，以吞吐量为主要评价指标，辅以 CPU 利用率分析。结果表明，本文方案在顺序 I/O 场景下能够提升 F2FS 的吞吐表现，且该收益可跨平台复现。下一章将对全文工作进行总结，并讨论当前方案的局限性与后续研究方向。
