## 图注

> 本图展示了基于 iomap 框架的 F2FS Large Folio 缓冲 I/O 总体架构。顶层是 iomap 框架中被 F2FS 复用的五个组件：①②③为三个入口框架，④⑤为两个通用数据结构/迭代器。F2FS 通过实现对应的 `iomap_begin` 回调嵌入自身逻辑，回调内部统一调用 `f2fs_set_iomap`（⑥）进行块级到区间级的映射转换，并通过 `f2fs_folio_state`（⑦）维护 per-block 状态位图和完成计数（⑧），共同构成被所有路径复用的核心机制层。GC 适配在复用 ⑥⑦⑧ 的基础上叠加子范围脏区标记和 bio 子页偏移修正；压缩读复用 ④（`iomap_iter`）并以自实现的 `f2fs_compress_readpage_iter` 复刻 ⑤（`iomap_readpage_iter`），同时扩展多簇物理块信息一次性获取能力；压缩写复用 ②（`iomap_file_buffered_write`）的 `f2fs_buffered_write_iomap_begin`，并扩展跳过完全覆盖中间簇的优化。所有路径最终经 bio 层统一提交到块设备。

## 与 v1 的关键差异

| 改动点 | v1 | v2 |
|--------|----|----|
| iomap 框架 | 散落在入口描述中 | 独立顶层框，编号 ①~⑤：三个入口框架 + `iomap_iter` + `iomap_readpage_iter` |
| 核心机制编号 | ①②③ | ⑥⑦⑧（iomap 层占用 ①~⑤） |
| 写回入口 | `iomap_writepages` | `writeback_iter`（实际代码使用的迭代器） |
| 压缩文件 | 单一方块 | 拆为"压缩读""压缩写"两个独立矩形 |
| 压缩读描述 | 无 | 复用 ④ + 复刻 ⑤ → `f2fs_compress_readpage_iter` |
| 压缩写描述 | 无 | 复用 ② + 扩展跳过全覆盖簇 |
| 核心机制⑧ | 仅 read/write pending | 加入 `dirty_bytes_pending`（压缩写回延迟解锁用） |
| 结构体名 | `f2fs_iomap_folio_state` | `f2fs_folio_state`（与实际代码一致） |
