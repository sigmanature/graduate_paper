1     项目概述
1.1   项目背景

    • Large Folios 的演进: large folios 是内核自从 5.15 版本起引入的对 page 结构体的替
      代, 同时也是内核对原有的复合大页 (Compound Page) 实现方案的全新升级, 由 Linux
      内存管理子系统维护者 Matthew Wilcox 先生发明。

       – 传统方案的痛点: 传统的复合页面依靠特殊的 flag 来判断头页面 (head page) 和尾
         页面 (tail page), 但是其操作接口全部都是基于 page 结构体。这会导致接口使用者
         感到迷惑, 因为仅从函数签名来看, 无法直观地区分一个 page 参数究竟应该传入头
         页面还是尾页面。
       – Folio 的革新: folio 的出现则彻底将接口清晰化和规范化。一个 folio 结构体自
         身即代表一个巨大的、完整的复合页面整体, 不再需要进行头尾页面的区分。同时,
         为了保持向后兼容, 原先基于头尾区分的 API 也可以轻松地将头页面指针类型转换
         为 folio 指针, 因为 folio 在内存布局上复用了头页面的位置。基于 folio 的引
         入, 可动态调节大小的大页面自此终于能广泛地在内核中得以使用, 原有的 4KB 小
         页面、固定大小死板僵硬的巨页 (HugeTLB)、以及使用场景比较局限的复合页面的
         尴尬局面被彻底打破。

    • iomap 框架的崛起: iomap 框架是 Linux 内核现在主推的新一代文件系统 I/O 框架。它
      基于迭代器 (iterator) 实现, 提供了一个现代化的、统一的文件系统 I/O 接口, 并且 iomap
      框架自身已经实现了文件系统所有的主要 I/O 路径。其它文件系统只需实现其暴露出来
      的核心钩子函数 (hook function), 如 iomap_begin 和 iomap_end, 就能以极为优雅的方
      式, 通过一个函数调用完成复杂的 I/O 工作。例如, 使用 iomap_readahead 完成缓冲读,
      或用 iomap_file_buffered_write 完成缓冲写。

    • f2fs 的现状与需求: f2fs (Flash-Friendly File System) 是当前安卓 (Android) 系统最广
      泛使用的文件系统。它因为其针对闪存 (Flash Memory) 友好的设计特性——例如通过日
      志结构 (Log-structured) 的设计来聚合写操作, 将随机写转换为顺序写, 从而有效减少写
      放大并延长闪存寿命——而广泛应用于手机这种使用闪存作为主要存储介质的移动式/嵌
      入式设备中。近年来, 随着硬件性能的提升和应用负载的加重,f2fs 社区也开始了对 large
      folios 的推进工作, 以期获得更好的性能提升。



1.2   项目意义
    为 f2fs 引入 Large Folios 与 iomap 支持, 不仅是顺应内核演进趋势的必要举措, 更是解锁下一
代存储性能、赋能前沿应用场景的关键一步。


    • 内核层面的性能收益:

       – Large Folios 的直接优势: 作为更大的内存管理单元,Folio 能够一次性覆盖更多的
         虚拟地址空间, 从而大幅降低因页表缓存 (TLB) 未命中而导致的性能损失。在内存
1   项目概述                                                                 4



          回收路径上, 由于 LRU 链表管理的内存单元数量急剧减少 (例如, 一个 64KB 的
          Folio 在 LRU 上只占一个节点, 而过去是 16 个), 扫描和迁移的成本显著降低, 系统
          在高内存压力下的响应能力得到增强。
        – iomap 带来的 I/O 效率革命: iomap 以“区间 (extent)”为单位进行批量处理, 彻底
          改变了传统文件系统“逐块 (block-by-block)”的 I/O 模式。在处理大文件时,iomap
          能够一次性完成大段连续逻辑块到物理块的映射, 并生成更少、更大的 bio 请求, 这
          极大地减少了元数据查找的开销和 I/O 提交的 CPU 消耗, 是实现高性能 I/O 的核
          心。

    • 对真实世界应用的深远影响:

        – 赋能 AI 与大数据: 在 AI 训练和推理场景中, 模型权重和海量训练数据的加载是典
          型的 I/O 密集型任务。更高的 I/O 吞吐和更低的 CPU 开销, 意味着可以更快地将
          数据送入计算单元, 缩短“data-to-compute”的延迟, 从而加速整个训练和推理流程。
        – 提升大型游戏与应用体验: 现代 3A 游戏和专业图形软件需要加载庞大的高清纹理、
          模型和场景数据。本项目带来的性能提升, 可以直接转化为更快的应用启动速度、更
          短的游戏关卡加载时间, 为用户提供无缝、流畅的沉浸式体验。
        – 优化数据库与数据中心负载: 对于数据库系统和云存储服务,I/O 延迟和吞吐量是核
          心性能指标 (KPI)。通过本项目技术, 可以有效降低读写延迟, 提升 IOPS 和带宽, 同
          时节省下来的 CPU 资源可以服务于更多的计算任务, 最终帮助数据中心降低总拥有
          成本 (TCO)。

    • 紧跟业界前沿技术浪潮:

        – 本项目所探索的技术是绝对的业界前沿。在移动端领域,Large Folios 带来的性能红
          利已成为各大厂商竞相追逐的焦点。目前, 国内如华为、OPPO、vivo 等领先的手机
          厂商均在其旗舰产品的操作系统内核中, 投入大量研发力量推进这项技术的适配与
          落地, 以构建其高端产品的核心技术壁垒。本项目的成功实践, 不仅与业界顶级厂商
          的技术路线保持一致, 也为该技术在更广泛场景下的应用积累了宝贵的实践经验。



1.3   项目预期目标与实际完成度

    • 预期目标与实际完成情况: 本项目的预期目标的原表述是能在 f2fs 普通文件的 I/O 路径
      上支持 large folios, 并让 f2fs 的垃圾回收机制能正确识别出 large folios, 最好能实现使用
      iomap 支持 large folios。

    • 超额完成的成果: 而我 ** 不但直接使用了 iomap 来完成 f2fs 文件的 I/O 工作, 还扩展
      了 iomap, 使得压缩文件也支持了 large folios, 同时还解决了 f2fs 支持 large folios 许多
      独有的难点, 属于超额完成了预期目标 **。



1.4   技术路线选型与核心挑战
    问题一：我们该如何为 f2fs 引入 Large Folios 支持？
1   项目概述                                                                  5


    理论上，文件系统可以直接在原始的 bufferedread/write 以及 pagewriteback 路径中，手动
添加对 LargeFolios 的支持。然而，经过我的深入调研，发现这几乎是一条不可行的道路。真正的
选择只有一个：拥抱 iomap 框架。并且，我很快意识到，仅仅“使用”iomap 是不够的，我们还必
须对其进行专有的扩展。


    为什么 iomap 是必由之路？ 因为它为我们解决了原生 f2fs 在支持 LargeFolios 时面临
    的一系列根本性难题：
    挑战 1: 性能瓶颈：逐块映射 vs. 区间映射

           • f2fs 原生方式: 以单个 Page (4KB) 为单位进行文件块地址的查找和 I/O 请求
             的构建，在处理大文件时存在大量的、重复的元数据查找开销。
           • iomap 解决方案: iomap 以区间 (extent) 为核心，能够一次性映射一段连
             续的逻辑块区间，并将其直接关联到覆盖该区间的整个 Folio。这在块映射和
             I/O 请求提交两个层面，都带来了数量级的性能提升。

    挑战 2: 写放大问题：模糊的脏页 vs. 精准的追踪

           • f2fs 原生方式: 对 LargeFolio 的脏页标记是“模糊”的，一旦 Folio 中有任
             何一个 Page 变脏，整个 Folio 都会被标记为脏。在回写时，这会导致大量未
             修改的干净数据被无效地写回，对闪存寿命造成严重影响，这就是“写放大”
             问题。
           • iomap 解决方案: 内核 6.6 版本为 iomap 引入了 iomap_folio_state 结构，
             它实现了逐块脏页追踪 (per-blockdirtytracking)。这使得我们能精准地知
             道一个 LargeFolio 中哪些块（而非整个 Folio）是脏的，从而只回写真正需
             要回写的数据，从根本上解决了写放大问题。

    挑战 3: 并发 I/O 问题：混乱的回调 vs. 原子计数

           • f2fs 原生方式: 其代码基于“一个 Page 等于一个块”的陈旧假设，完全没有
             考虑一个大的 Folio 内部可能包含多个并发 I/O 的情况。这会导致 bio 的结
             束回调函数 end_io 被过早调用，错误地解锁 Folio 或结束其 writeback 状
             态，引发数据竞争和一致性问题。
           • iomap 解决方案: iomap_folio_state 中的 read_bytes_pending 和 writ
             e_bytes_pending 原子计数器，正是为解决此问题而生。它能精确记录一个
             Folio 中还有多少字节的数据“在途”
                                ，只有当所有 I/O 都完成，计数器归零
             时，才会执行最后的回调操作。

    挑战 4: 生命周期管理：繁琐的手动 vs. 优雅的自动

           • f2fs 原生方式: 其对 Folio（或 Page）的加锁、解锁、标记状态等生命周期
             管理逻辑非常繁琐，且只具备处理 0 阶 Folio 的意识，直接改造以支持高阶
             Folio 的成本极高。
           • iomap 解决方案: iomap 框架本身没有任何历史包袱，其设计原生支持高阶
             Folio，并能自动、优雅地处理其复杂的生命周期管理。
2   背景知识和数据结构介绍                                                     6


    问题二：为什么仅仅使用 iomap 还不够？ 在将 iomap 用于 f2fs 的过程中，我很快发现，虽
然 iomap 解决了通用性的问题，但 f2fs 作为一个特性丰富的文件系统，其“特殊性”是通用框架
无法完全覆盖的。这引出了我设计的核心：对 iomap 进行扩展和适配。具体来说，挑战主要来源于
以下几点：


    挑战 1: 压缩文件的特殊性: f2fs 支持文件压缩，这意味着内存中存在两种页：原始页和
           压缩页。压缩页位于一个特殊的 COMPRESS_MAPPING 缓存中，并且其真正落盘与
           否、与 I/O 实体 bio 的映射关系，都与普通数据页完全不同。而通用的 iomap 对
           address_mapping 的机制完全不同，无法直接处理。

    挑战 2: folio->private 字段的冲突: 我调研发现，f2fs 会司空见惯地将特有的私有标志
           位（如 FI_ATOMIC_FILE）存储在 page->private 字段中。当 Page 被整合为
           Folio 后，folio->private 继承了该字段。这与 iomap 期望用该字段存储 io
           map_folio_state 指针的行为产生了直接冲突。为了解决这个问题，我设计了
           f2fs_iomap_folio_state 结构体，它巧妙地扩展了 iomap_folio_state，为
           其增加了一个存储 f2fs 私有标志的空间。

    挑战 3: 日志结构文件系统的本质: f2fs 是一个日志结构文件系统，其大量的写入操作是
          “异地更新”(out-of-place)。这意味着在脏页回写时，我们写入的是一个新的物
           理块地址，而原始数据块的地址没有任何意义。这使得 iomap 框架中依赖于稳定块
           映射的 f2fs_map_blocks 函数几乎无法编写。

    挑战 4: Checkpoint 与重试机制: f2fs 依赖 Checkpoint 机制来保证文件系统的一致性。
           当 Checkpoint 进程启动时，它会阻塞所有新的文件操作。因此，f2fs 的脏页回
           写路径必须能够优雅地处理这种情况，支持可重试的回写 (retryablewriteback)。
           然而，通用的 iomap 框架并未提供文件系统定制该重试逻辑的接口。


    综上所述，我意识到，要真正实现一个健壮、高效的 LargeFolios 支持，我必须化用 iomap
    的核心功能和思想，重写针对 f2fs 压缩文件的 bufferedread 和 pagewriteback��。

2.1     通用背景知识
2.1.1   Linux 文件系统的四层抽象模型

    要理解 iomap 和 folio 带来的改进, 我们首先需要建立一个关于文件在 Linux 内核中如何被看
待的四层抽象模型。这套模型精妙地解耦了不同层级的关注点。可以将其类比于计算机网络的协议
栈, 每一层都只关注自己视角的索引单元, 并定义了如何将自己的单元转换 (或映射) 到下一层的单
元, 对其他层的复杂性保持透明。


        1. 用户视角 (User Space): 用户日常接触到的文件, 其内容看起来是连续的。当我们使
         用 read() 或 write() 系统调用时, 我们提供的是一个文件内的字节偏移量 loff_t
2   背景知识和数据结构介绍                                                          7



          pos 和一个长度 size_t len。这一层操作的是逻辑上连续的字节流。

        2. 内存视角 (Memory Layer): 内核为了加速访问, 会将文件的内容缓存到内存中。在这
          一层, 文件的字节流被分割成固定大小的块, 即页 (Page) 或更大的页 (Folio)。这一层
          通过页索引 (pgoff_t) 来寻址。关键的转换公式是: pgoff_t = pos / PAGE_SIZE。

        3. 文件系统视角 (Filesystem Layer): 文件系统自身有一套逻辑块 (FS Blocks) 的概
          念, 大小通常为 4KB。这一层维护着核心的元数据, 负责将内存中的页索引映射到文件
          系统自己的逻辑块号。在页大小等于文件系统块大小的情况下, 这个转换是一对一的:
          fs_block_no=block_mapping(index)。

        4. 块设备视角 (BIO Layer): 这是最底层的 I/O 抽象。文件系统的逻辑块号最终需要
          被转换为块设备驱动能理解的地址。这一层将文件系统逻辑块号转换为逻辑扇区地址
          (LBA), 并封装成 bio 结构体提交给块设备驱动。需要注意的是, 这里的” 逻辑扇区”
          并非磁盘硬件上的真实物理扇区, 而是操作系统构建的一个虚拟的、连续的扇区地址空
          间。转换公式通常是: LBA=fs_block_no*FS_BLOCK_SIZE。


    完整的转换示例: 假设用户要读取文件的第 20480 个字节开始的 8192 个字节 (即 2 个页), 在一
个页大小为 4KB、文件系统块大小为 4KB、扇区大小为 512B 的系统中:


        • 用户层: loff_t pos = 20480, size_t len = 8192

        • 内存层: pgoff_t index =20480/4096=5, 即从第 5 个页开始, 需要读取 2 个页 (页 5
          和页 6)。

        • 文件系统层: 对应文件系统逻辑块号 5 和 6。文件系统通过其元数据 (如 extent tree)
          查找这两个逻辑块在磁盘上的物理位置, 假设它们被映射到物理块号 1000 和 1001。

        • BIO 层: 物理块号 1000 对应的 LBA 为 1000*4096=4096000, 物理块号 1001 对应的
          LBA 为 1001*4096=4100096。最终生成的 bio 请求会将区间 [4096000,4100096+40
          96-512] 提交给块设备驱动。


    这种分层的核心在于, 每一层都有自己的” 视角协议”: 用户层看到的是字节偏移, 内存层看到的
是页索引, 文件系统层看到的是逻辑块号, 而 BIO 层看到的是逻辑扇区地址。每一层只需要知道如
何与上下相邻层进行转换, 无需了解其他层的复杂性。

2.2.2    folio 基本背景与支持挑战

     从四层模型视角来看, folio 和 page 一样, 都是使用 pgoff_t 作为索引单元。因此, 似乎直观上
来看, 我们对于高阶 folio, 也可以将其抽象成单个单个的 pgoff_t 索引, 然后逐个和 fs_block_no
进行映射。
     然而, 结合 folio 的设计来看, 会发现有很大的问题。首先 folio 逻辑上表示一整个大页。这
意味着所有对一块内存元数据的各种操作, 比如锁、引用计数, 都是针对 folio 来进行的。如果将其
拆成按照逐个 pgoff_t 索引的 page, 那 folio 的整体性信息就丢失了。并且我们将很难进行引用
计数和上锁的维护。从 folio 拆分成的多个 page, 它们的头 page 都将指向 folio。这意味着如果
说我们要进行上锁等各种操作, 我们只能使用老式的 page 相关接口。这将增加上锁解锁和引用计数
的使用次数。使用 folio 相关接口几乎是一定会造成多重上锁/解锁这样的 bug 的。这和 folio 的
设计哲学完全是背道而驰的。
     那么, 我们最好从代码的概念上, 就应该抛弃对 folio 按逐个 pgoff_t 索引的设计思想。转而
一次将一个 folio 视为一个具有起始 pgoff_t 和以页单位为长度的整体。而 folio 最大的特性是
可以变化 order, 也就是变化它的页面数量。这种情况下, 我们该怎么解决支持 largefolios 的挑
战呢?

2.2.3    iomap 迭代框架：支持任意变长 folios 的关键

     iomap 框架通过其巧妙的迭代器设计，完美解决了支持任意变长 folios 的挑战。其核心思想
是将 ** 区间映射 ** 与 ** 迭代处理 ** 相结合，使得无论 folio 的阶数如何变化，都能以统一的
方式处理。
     iomap 结构体承上而启下，它同时存储了内存中连续区间信息和磁盘 LBA 地址字节映射：
2   背景知识和数据结构介绍                                                                    9


    iomap 结构体的区间映射能力

    struct iomap {
          u64 addr;                 // 映射到的磁盘字节偏移量 (支持大区间)
          loff_t offset;            // 映射在文件中的字节偏移量
          u64 length;               // 映射的字节长度 (关键：支持任意长度)
          u16 type;                 // 映射类型 (IOMAP_MAPPED, IOMAP_HOLE, ...)
          /∗ ... other members ... ∗/
    };



        这种以 ** 字节为粒度的区间描述 ** 使得 iomap 能够天然地适配任何大小的 folio。无论是
4KB 的 0 阶 folio，还是 64KB 的 4 阶 folio，iomap 都能通过其 length 字段准确描述整个区间
的映射信息。
        iomap 的迭代器 iomap_iter 实现了这一切的自动化处理：

    iomap_iter 的核心迭代逻辑

    int iomap_iter(struct iomap_iter ∗iter, const struct iomap_ops ∗ops)
    {
          // 检查是否需要开始新的映射
          if (!iter−>iomap.length)
                 goto begin;


          // 计算已处理的字节数
          advanced = iter−>pos − iter−>iter_start_pos;
          olen = iter−>len + advanced;


          // 调用文件系统的结束回调（如果需要）
          if (ops−>iomap_end) {
                 ret = ops−>iomap_end(iter−>inode, iter−>iter_start_pos,
                            iomap_length_trim(iter, iter−>iter_start_pos, olen),
                            advanced, iter−>flags, &iter−>iomap);
                 if (ret < 0 && !advanced)
                     return ret;
          }


          // 判断是否继续迭代
          if (iter−>status < 0)
                 ret = iter−>status;
          else if (iter−>len == 0 || (!advanced && !stale))
                 ret = 0;     // 完成迭代
          else
                 ret = 1;     // 继续迭代


          // 重置 iomap 状态，准备下次映射
          iomap_iter_reset_iomap(iter);
          if (ret <= 0)
2   背景知识和数据结构介绍                                                                    10



              return ret;


    begin:
          // 调用文件系统的开始回调，获取新的映射
          ret = ops−>iomap_begin(iter−>inode, iter−>pos, iter−>len, iter−>flags,
                            &iter−>iomap, &iter−>srcmap);
          if (ret < 0)
              return ret;
          iomap_iter_done(iter);     // 更新迭代器状态
          return 1;
    }



        这种设计的核心优势在于 ** 自适应的区间处理能力 **：


         1. 统一的接口: 上层调用者只需编写特定功能的 iomap_begin 函数将映射信息放入 io
           map 结构体中返回，然后使用简洁的 while(iomap_iter) + 内部执行函数的形式就
           能完成所有工作。

         2. 自动区间前进: iomap_iter 会根据每次处理的字节数自动前推 iter.pos，无论每次
           处理的是 4KB 还是 64KB 的区间。

         3. 变长 folio 适配: 当遇到高阶 folio 时，文件系统的 iomap_begin 可以返回一个覆盖
           整个 folio 的大区间映射，迭代器会自动处理这个大区间，避免了将 folio 拆分成多
           个 page 的复杂操作。

         4. 性能优化: 对于连续的磁盘块，可以在一次 iomap_begin 调用中返回一个大区间，减
           少了反复调用的开销。


        例如，在处理一个 16KB 的 2 阶 folio 时：- 传统方法：需要 4 次独立的页映射调用 - iomap
方法：一次 iomap_begin 调用返回 16KB 的区间映射，迭代器自动处理整个区间
        这种 ** 区间化的思想 ** 使得 iomap 框架能够无缝支持从传统的 4KB 页到未来可能出现的更
大的 folios，为文件系统的性能优化提供了坚实的基础。

         并且旧的 fio->page 接口已被废弃，为了支持 largefolios，我们现在必须追踪一个
         folio 的哪个子部分正在被提交到 bio 中。

     • 新增 idx 和 cnt 字段: 为了实现这一点，我们向 fio 结构体中添加了 idx 和 cnt 两个
         字段。

          – fio->idx: 代表本次 I/O 操作在当前 folio 内的起始偏移量（以页为单位）。
          – fio->cnt: 代表本次 I/O 操作需要处理的连续块的数量。

     • 重释地址字段: 随着这两个字段的引入，fio 中现有的 old_blkaddr 和 new_blkaddr
3   逐功能深度解析                                                                  13



        字段的含义被重新解释。它们现在代表与 fio->idx 相对应的 起始新旧块地址。因此，
        一个 fio 不再代表从一个 old_blkaddr 到一个 new_blkaddr 的单一映射，而是代
        表一个范围映射：从块地址范围 [old_blkaddr,old_blkaddr+fio->cnt-1] 映射到
        [new_blkaddr,new_blkaddr+fio->cnt-1]。

    • 确保向后兼容: 在 bio 提交路径中，对于 fio->cnt 未被显式初始化的情况，我们将其
        默认设置为 1，并将 fio->idx 默认设置为 0。这确保了与所有现存的、仅在单个页面上
        操作的 f2fs 逻辑的向后兼容性。


2.2.6   数据结构的转换流程

    这些结构体构成了从 f2fs 内部缓存到通用 I/O 框架的转换链条, 其核心逻辑关系如下:


        1. Extent Cache 命中: 当 f2fs 需要进行块映射时, 首先会使用请求的逻辑块区间查询
          extent cache。如果命中, 一个 extent_info 结构体会被找到。

        2. extent_info → f2fs_map_blocks: f2fs 会将 extent_info 中的信息 (以块为单位的
          fofs, len, blk) 直接转换为 f2fs_map_blocks 结构体中的 m_lblk, m_len, m_pblk。
          这一步是在文件系统内部完成的, 单位仍然是 ** 文件系统块 **。

        3. f2fs_map_blocks → iomap: 这是最关键的转换。iomap 框架调用文件系统的 ma
          p_blocks 接口 (在 f2fs 中即 f2fs_map_blocks 函数) 后, 会得到一个填充好的
          f2fs_map_blocks 结构。随后, 通过 f2fs_set_iomap 函数将这个以 ** 块 ** 为单位
          的结果, 转换为通用的、以 ** 字节 ** 为单位的 iomap 结构, 从而建立起文件逻辑字
          节区间到磁盘 LBA 区间的完整映射。最终, 上层的 I/O 逻辑 (如 iomap_readahead)
          便可直接使用这个与具体文件系统无关的 iomap 结构来处理数据。


    这个流程清晰地展示了 iomap 如何通过一层抽象, 将 f2fs 特有的、基于块的 extent_info 和
f2fs_map_blocks, 优雅地转换为通用的、基于字节的 I/O 描述, 从而实现了文件系统与 I/O 逻辑
的解耦。

3.1.1   功能背景

    普通文件的 buffered read 是文件系统中最基础和最频繁的操作之一。传统的 f2fs 实现采用逐

过 iomap 的区间化处理思想，我们能够一次性处理大片连续的内存区间和磁盘区间，从而完美支持
任意阶数的 largefolios。

3.2     普通/原子文件 buffered write
3.2.1   功能背景

    在我们深入代码细节之前，必须明确一个良好的缓冲写入（Buffered Write）机制需要实现的核
心目标。这不仅是所有文件系统的基础功能，也是理解 iomap 框架带来优越性的关键。


        1. 支持文件创建与顺序追加 (Append): 我们来思考一下, 很多时候我们 touch 了一个新
         的文件, 然后, 我们可能用 vim 或者 echo 了很多文本进去。这个时候我们 vim 保存
         或者 cat 一下, 是能看到文件内容的。但是, 我们知道,touch 创建的新文件, 其内容是
         空的。或者, 我们往 vim 里追加了一段文字, 但是我们知道, 原本的文件没有这段内容,
         这点可以通过看之前的文件大小进行验证。这同时意味着, 磁盘上原先是没有这段内
         容的。但是对于用户来说, 我们就是能正常看到这段内容。这就是 buffer write 需要满
         足的第一个目标, 能支持文件的顺序追加的写入, 同时扩展文件的大小。当用户创建一
         个新文件并写入内容，或者在一个已有文件的末尾追加内容时，文件系统必须能够：

          • 在逻辑上文件末尾的位置，按需分配新的物理块。
          • 将用户数据写入到内存中的页缓存 (Page Cache / Folio)。
          • 更新文件大小元数据。
          • 最终由后台进程将这些“脏”页回写到磁盘上新分配的位置。

         对用户而言，这个过程是透明的，他们只看到文件变长了，内容增加了。

        2. 支持文件覆盖写 (Overwrite): 好, 现在我们的文件有内容了, 我们重启了一下系统, 此
         时我们想编辑一下文件内容, 很容易, 打开 vim 然后到我们想改的地方改一下, 立刻
         就能看到效果。但是我们细想一下, 因为我们重启过系统, 所以一开始内存中是没有
         缓存的文件数据的。那我们是怎样将文件内容精确写到正确的位置的呢？毕竟我们在
         buffered write 的场景下不能直接对着磁盘写入对吧。所以这时候 buffered write 要满
         足的第二个目标是同时要触发 buffered read, 将文件内容读到内存中, 然后我们修改内
         存中的内容, 最后将内存中的内容写入磁盘。因此，文件系统必须：

          • 首先，像读操作一样，找到待修改数据所在的物理块。
          • 将这部分数据从磁盘读入页缓存（如果尚未缓存）。这个过程通常被称为“Read-
            Modify-Write”。
          • 用户在内存中修改数据。
          • 将修改后的脏页标记起来，等待回写。


这两个目标，特别是它们对块映射（找到已有块或分配新块）的依赖，是传统实现与 iomap 实现产
生巨大差异的核心所在。接下来，我们将围绕这两个目标，逐一剖析 F2FS 在引入 iomap 前后的实
现差异。
3   逐功能深度解析                                                                   28

3.3       f2fs_iomap_folio_state
3.3.1     功能背景

        在 Linux 内核的传统设计中，page->private (现在是 folio->private) 字段是一个 void* 类
型的指针, 被文件系统广泛用于存储与特定内存页相关的私有状态。F2FS 也不例外，它利用这个字
3   逐功能深度解析                                                             38


段的各个比特位来存储一系列标志，如 PAGE_PRIVATE_ATOMIC_WRITE（原子写）
                                               、PAGE_PRIVATE_O
NGOING_MIGRATION(GC 迁移中) 等。
     然而，当 iomap 框架引入后，它也需要使用 folio->private 来存储一个指向 iomap_folio_
state 结构体的指针。这个结构体对于实现高阶 Folio 的逐块脏状态跟踪和并发 I/O 控制至关重要。


     核心冲突 (The Core Conflict)
        • F2FS 的需求: 需要在 folio->private 中存储多个布尔型标志位。

        • iomap 的需求: 需要在 folio->private 中存储一个指向 iomap_folio_state 结构
         体的指针。


这两者在 folio->private 这个有限的资源上产生了直接冲突。更复杂的是，我们为 F2FS 压缩文
件设计的 cc_bytes_pending 计数器也需要一个存储位置。我们不能简单地抛弃任何一方，必须找
到一种方法，将 F2FS 的私有状态与 iomap 的状态跟踪机制融合到一个统一的结构中。

3.3.2   重大问题分析

     我们已经在问题背景中分析过 f2fs_iomap_folio_state 的必要性。若没有它，导致的直接
问题现象，就是在任何需要使用 f2fs_iomap_folio_state 的地方，以及 iomap 框架中需要访问
iomap_folio_state 的代码，会将其（folio->private 中存储的 F2FS 标志位）误认为是一个指
针值，导致内核在形式上抛出空指针解引用的异常。
     当我们将 f2fs_iomap_folio_state 集成到代码中时，我发现真是牵一发而动全身。任何原先
使用旧的 set_page_private/get_page_private 的代码路径全部都需要更改。这里面涉及到的代
码路径不光有普通数据 folio，还有元数据 folio 中存储自己 private 字段的地方也需要替换成我们新
设计的接口。

f2fs_iomap_folio_state 的诞生         经过 large folios 创始人 Matthew Wilcox 先生的指点, 我决定
创建一个 F2FS 专属的、与 iomap_folio_state 兼容但功能针对 F2FS 需求进行了扩展的结构体：
f2fs_iomap_folio_state。

这个设计的精妙之处在于：


       1. 内存布局兼容性: 它的前三个成员（state_lock, read_bytes_pending, write_byte
         s_pending）与 iomap_folio_state 的定义完全一致。
         这意味着，任何操作 iomap_folio_state 的通用 iomap 函数，都可以无缝地操作一

       2. 柔性数组扩展: 我们利用 C 语言的柔性数组成员 state[],
         在 iomap_folio_state 所需的内存空间之后，额外分配了两个 unsignedlong 的空

          • 第二个额外空间被用作我们为压缩文件设计的 cc_bytes_pending 原子计数器。


     核心实现: 智能分发的 Folio 私有数据 API 为了取代旧的 page_private 宏，我们设计了一
整套新的宏，用于生成对 folio->private 进行获取（Get）
                                   、设置（Set）和清除（Clear）操作的函

     我们利用了 F2FS 原有标志位设计中的 PAGE_PRIVATE_NOT_POINTER 位，它天生就能区分
folio->private 中存储的是“标志位 **”还是一个“指针”
                                   。我们的 f2fs_iomap_folio_state 恰
好就是一个指针，这使得新旧两种模式可以完美共存和区分。
     Get/Clear 操作的多级判断逻辑: 对于获取（Get）或清除（Clear）标志位的操作，函数会首先
直接测试 PAGE_PRIVATE_NOT_POINTER 标志位。

    • 如果标志位被设置: 说明 folio->private 存储的是旧式的标志位。函数会直接在此值上进行
     位操作并返回。这是一个高效的快速路径，完美兼容了元数据 Folio 或其他不与 iomap 交互的
     场景。

    • 如果标志位未被设置: 说明 folio->private 是一个指针（或为 NULL）。函数则会继续检查该
     指针指向的结构体是否包含我们的魔数 F2FS_IFS_MAGIC。如果匹配，就在 f2fs_iomap_fol
     io_state 的扩展区域进行操作。

     Set 操作的智能分配逻辑: 设置（Set）操作的逻辑更为精妙，它需要决定是否要分配 f2fs_iom
ap_folio_state 结构体。

    • 它会先通过 f2fs_should_use_buffered_iomap 判断当前 Folio 是否可能进入 iomap 的缓冲
     写路径。

    • 如果不会进入，且 folio->private 当前为空，那么它会沿用旧逻辑，直接在 folio->private
     中设置标志位。

    • 但如果会进入 iomap 路径（由 force_alloc 参数体现）
                                        ，或者 folio->private 已经有值（可
     能是标准的 iomap_folio_state），那么它必须走 f2fs_ifs_alloc 的完整路径。

3.4       压缩文件 buffered read
3.4.1    功能背景

     F2FS 作为专为闪存设备设计的文件系统, 其压缩功能对于节省存储空间至关重要。然而,F2FS
压缩文件的读取流程与普通文件截然不同。当内核需要读取一个被压缩的簇时，它不能直接将磁盘
上的数据读入目标 Folio，而必须执行一个多阶段的“间接读取”流程：


         1. 读取到临时缓冲区：先将压缩数据块读入由 compress_page_pool 分配的缓冲页中。

         2. 解压：CPU 调用解压算法（如 lzo, lz4, zstd 等），将数据解压到目标 Folio 中。


     这个流程意味着，通用的 iomap_readahead 假设的“磁盘到 Folio”的直接数据通路在此失效。
更复杂的是，F2FS 文件支持部分压缩，即一个文件内可能同时包含压缩数据和非压缩数据。当引
入 Large Folios 机制后，一个大的 Folio 完全可能跨越一个或多个压缩簇以及非压缩数据块，这就
对 I/O 的管理和并发控制提出了严峻的挑战。

3.4.2    重大问题分析

     在为 F2FS 的压缩读路径引入 Large Folios 支持时，核心挑战源于 **I/O 模型的混合性与状态
管理的复杂性 **。一个 Large Folio 的不同部分可能需要通过完全不同的路径来填充数据，导致了
以下关键问题：

    • 混合 I/O 模型: Folio 的一部分可能通过标准的直接 I/O 路径完成（非压缩部分），而另一部
        分则需要走“读取到临时页 -> 解压”的间接路径（压缩部分）。

    • 并发与状态不一致: 这两条路径的 I/O 完成时机完全不同。如果不能统一管理整个 Folio 的完
        成状态，就可能出现一个线程已经完成了非压缩部分的读取，而另一个解压任务仍在进行中。
        此时若错误地解锁 Folio，就会导致数据损坏。

    • 生命周期追踪: 必须有一个精确的机制来追踪一个 Large Folio 上所有悬而未决（pending）的
        I/O 请求。只有当所有相关的 I/O 操作（无论是直接读还是解压）都成功完成后，才能将整个
        Folio 标记为 Uptodate 并解锁。

     为解决此问题，我们引入了 read_bytes_pending 计数器机制，对一个 Large Folio 发起的所
有 I/O 请求进行统一追踪，确保其生命周期管理的正确性。
3   逐功能深度解析                                                                     53

3.5     压缩文件 buffered write
3.5.1   功能背景

    我们已经决定将整个 bufferedwrite 路径迁移往 iomap 框架，因为经过前面的分析我们知道，
这是最无历史包袱、能够原生支持 largefolios 的方式。
    与 bufferedread 和 pagewriteback 不同的一点是，iomap_buffered_write 除了提供通用的
iomap_begin 和 iomap_end 接口外，还允许文件系统通过 `folio_ops` 回调函数集，提供自己定
制化的 folio 获取逻辑 (__iomap_get_folio) 和释放逻辑 (__iomap_put_folio)。通过分析，我
发现利用 iomap_buffered_write 框架来实现 F2FS 压缩文件的缓冲写是完全可能的，并且其函数
调用栈可以做到比原有实现更少的开销。

3.5.2   重大问题分析

    初赛方案的局限性
    在初赛时，我们的设计方案是将我们自己改造成支持 largefolios 的 prepare_write_begin
逻辑（即原先 F2FS 的 bufferedwrite 逻辑）直接拿过来填入到 iomap_begin 之中。然后，在
__iomap_get_folio 中，我们和原代码逻辑一样，从压缩文件的上下文（compresscontext）中根
据文件内偏移量取出对应的 page，并将其转化为 folio 结构体。
    这种方法的根本问题在于, 我们的操作视角被局限在了单个“压缩簇”(compressioncluster)
内。这意味着，即使上层 VFS 希望写入一个横跨多个簇的大块数据，我们的 iomap_begin 和 __io
map_get_folio 也只能以簇为单位、一次只处理一个簇大小的 folio。
    这种限制的根源在于 VFS/MM 层的核心 folio 分配策略。当 iomap_get_folio 调用底层的
__filemap_get_folio 时，虽然它会尝试分配一个较大阶数（order）的 folio，但一旦失败，它
会立刻降低阶数并重试。

3.5.3    方案设计

        为了克服初赛方案的瓶颈，我们设计了一套全新的算法。其核心思想是 ** 切换视角 **：我们
不再是以“压缩簇”的视角出发，一次次地用小 folio 去填充一个簇；而是 ** 从一个“大 Folio”

磁盘读取覆盖这 128KB 范围所需的多个（可能是 4 个 32KB 的）压缩簇数据，一次性填满这个大
folio。这与旧算法一次只处理一个簇的模式形成了鲜明对比。这是我们的原创核心算法。

这个新算法的实现流程和关键设计点如下：


        1. 准备阶段 (iomap_begin):

                 • 分配大 Folio: 我们首先根据用户请求的写入位置和长度，计算出需要覆盖的所
                  有压缩簇的起止范围，并以此为据，调用 iomap_get_folio 一次性分配一个能够
                  容纳所有数据的、已锁定的 largefolio。
3   逐功能深度解析                                                                  66



           • 填充 Folio: 我们启动 F2FS 异步的 I/O 逻辑 (do_read_multi_folios)，提交一
             个或多个 BIO，从磁盘读取数据来填充这个 largefolio。在循环中，我们根据
             实际读取的簇长度，累加更新 iomap->length，确保 iomap 框架能正确认知本次
             操作的范围。
           • 传递 Folio: 当 Folio 被完全填充并处于 up-to-date 状态后，我们并不立即解锁
             它，而是将其指针保存在 iomap->private 字段中，传递给 iomap 的核心处理逻
             辑。

        2. 同步等待机制: F2FS 内提交 I/O 的代码全部是异步的。但 iomap_buffered_write
         必须等到整个 folio 的内容都从磁盘读上来（即完全 up-to-date）后，才能对其进
         行写入，否则会发生严重的数据不一致问题。内核虽提供了 submit_bio_wait，但它
         只能等待单个 BIO，无法满足等待一个可能横跨多个 BIO 的 largefolio 的需求。

           • 巧用 read_bytes_pending: 我们利用了 f2fs_iomap_folio_state 中的 read_b
             ytes_pending 字段。在提交 BIO 前，我们先给它加上一个偏置（bias）。这个
             偏置会阻止 I/O 完成回调函数过早地唤醒其他等待者。
           • 显式轮询: 在 iomap_begin 中，我们显式地循环等待，直到 read_bytes_pending
             的值变为 F2FS_IFS_MAGIC+1（我们的魔数加上偏置）

             提交的 BIO 都已完成。此时，folio 必然是完全 up-to-date 的。
           • 这种“加偏置-轮询-去偏置”的模式，为我们实现了一个高效、可靠的、folio 粒
             度的同步等待机制。

        3. 簇对齐与一致性: 为了避免一个压缩簇中只有部分 folio 处于 up-to-date 的尴尬情
         况（特别是当 folio 本身大小不足以覆盖整个簇时）
                                   ，我们的代码还会显式地将 folio
         最后一个页面所在的整个簇全部读入，确保了数据的一致性。

        4. 执行阶段 (__iomap_get_folio): 我们的 f2fs_iomap_get_folio 实现被大大简化。

3.6     脏页写回 Page Writeback
3.6.1   功能背景

    在深入探讨脏页回写之前，我们必须首先理解 F2FS 压缩文件的工作机制。F2FS 并非逐块压缩，
而是将多个连续的逻辑块（通常是 4 个，即 16 KB）组成一个压缩簇 (CompressCluster)。文件系
统会尝试将整个簇作为一个单元进行压缩。如果压缩后的体积小于原始体积，F2FS 就会将这个压缩
后的数据块作为一个整体写入磁盘，并在磁盘上将簇的第一个块地址用一个特殊的 COMPRESS_ADDR
标记来指代这一个簇的起始。
3   逐功能深度解析                                                     67



    核心概念: 压缩簇 (CompressCluster)
        • 原子单元: 压缩和解压总是以整个簇为单位进行。要读取簇中的任何一页，都必须先将
         整个簇读入内存并解压。

        • 延迟提交: 当写入一个压缩文件时，脏页不会立即被提交到bio 层。它们会被暂存到
         一个名为 compress_ctx 的压缩上下文中。只有当这个上下文被一个完整的簇填满时，
         F2FS 才会触发压缩和回写操作。


3.6.2   核心挑战与关键问题

    f2fs 的压缩簇目前的实现是一个簇中的page 数量是固定的4<<n, 默认情况下n=1。然而, 从内
存中分配的largefolio 阶数是无法预知的,(内存层有一套自适应阶数分配算法)。因此我们不能假
设一个folio 的整体处于同一个簇内。这带来了第一组严峻的挑战：


    固有的架构性挑战
        • 多对多关系: 一个高阶 Folio（如 64 KB）可能横跨多个压缩簇。反之，一个压缩簇
        （如 16 KB）也可能只包含一个高阶 Folio 的一部分。这种交错关系使得追踪和管理
         变得异常困难。

        • 生命周期不匹配: Folio 的锁和 writeback 状态是针对整个大内存块的。但它的不同
         部分可能属于不同的压缩簇，这些簇的回写时机是完全独立的。如何在一个 Folio 的
         一部分还在压缩上下文中等待时，正确处理另一部分的回写完成？

        • 并发与截断风险: 在旧的重试逻辑中，对 Folio 的反复加锁和解锁，为并发的部分截
         断（PartialTruncation）创造了巨大的风险窗口。一个被部分截断的 Folio 状态已
         变，但旧的检测机制无法识别，继续写入可能导致数据损坏。

        • 重试逻辑失效: 当写回因锁竞争而返回 -EAGAIN 时，我们必须能从上次失败的精确字
         节位置重试，而不是重写整个 Folio，以避免灾难性的写放大和数据不一致。对于f2fs,
         因为它异地分配新数据块写入的特性, 重复写回folio 的同一部分几乎一定会造成磁
         盘上的数据混乱。


在上述固有挑战的基础上，我们进一步发现了由内核回写机制和上游补丁引入的更细微、但同样致
命的问题：


    决赛期间发现的新增挑战
        • 后台回写的“空悬”Folio: 内核的后台回写线程（如 pflush）会定期扫描并回写脏
         页。在它的扫描批次中，可能包含大量已被清理脏位但仍处于回写状态的 Folio。这些
         Folio 不会进入我们真正的 I/O 提交流程（即find_dirty_range找不到脏范围）
                                                         ，因此
         它们不会被正常解锁，也不会通过 bio 回调来结束回写状态（folio_end_writeback）
                                                          ，
         导致其永久“挂起”，阻塞后续操作。

        • 子页索引错误: 在 f2fs_do_write_data_page 中，一个考虑欠妥的上游改动在查询块
3   逐功能深度解析                                                                        68



              地址时，错误地使用了 folio->index（Folio 首页的索引），而不是当前正在处理的
              子页的正确索引。这导致对 LargeFolio 的原地更新（in-placeupdate）逻辑彻底失
              效。

         • Bio 提交流程错误: 在 f2fs_submit_page_bio 中，另一个上游改动在将 Folio 的一
              部分添加到 bio 时，错误地将长度设置为整个 folio_size，而不是单个 PAGE_SIZE。

3.7       垃圾回收 GC
3.7.1    功能背景


        闪存的物理特性与挑战
        • 写入限制: NAND 闪存设备有一个根本性的物理限制：在向一个存储单元（页，Page）写
         入新数据之前，必须先擦除其所在的整个块（Erase Block）
                                        。一个擦除块通常包含数百个
         页。

        • 写放大 (Write Amplification): 如果文件系统天真地在原地更新数据（例如，只修改一
         个 4KB 文件）
                 ，它将不得不读取整个擦除块（如 2MB）到内存，修改那 4KB 数据，然后
         擦除整个块，最后再将整个 2MB 数据写回。这个过程导致实际写入磁盘的数据量远大于
         用户请求的数据量，即” 写放大”。这会急剧消耗闪存的有限擦写寿命（P/E Cycles）并
         降低性能。




        F2FS 的日志结构设计与 GC 的诞生

        • 日志结构文件系统 (LFS): 为了克服上述挑战，F2FS 采用了日志结构（Log-structured）
         的设计。它从不原地更新数据，而是将所有修改（包括新数据和旧数据的更新版本）作
3   逐功能深度解析                                                                80



        为” 日志” 顺序追加到闪存的干净区域。这种方式将大量随机的小 IO 聚合成连续的顺序
        大 IO，极大地提升了写入性能并保护了闪存寿命。

    • ” 无效数据” 的产生: 这种设计的副作用是，当文件被修改或删除时，旧版本的数据块并
        不会被立即擦除，它们只是在文件系统的元数据中被标记为” 无效”。随着时间的推移，文
        件系统中会散布着大量不再被任何活动文件引用的无效数据块，占用了宝贵的存储空间。

    • 垃圾回收 (Garbage Collection) 的使命: GC 机制就是为了解决这个问题而生的。它的
        核心任务是定期扫描文件系统，找到那些混合了有效和无效数据的” 脏” 段（Segment）。
        然后，它会将其中仍然有效的活数据块（Live Blocks）复制（搬运）到一个新的、干净的
        段中，并更新元数据指向这些新位置。一旦所有活数据都被搬走，原来的” 脏” 段就只剩
        下无效数据，可以被安全地整个擦除，从而回收空间以备后续写入。可以说，GC 是 F2FS
        维持长期稳定性能和可用空间的生命线。


3.7.2   重大问题分析

    在主线内核中，Matthew Wilcox 先生的补丁已经将 GC 的接口从 page 迁移到了 folio。然
而，这种迁移是初步的，其实现仍然基于” 一个 folio 等于一个 4KB 页” 的陈旧假设，这在引入
largefolios 后引发了两个严重问题。


    问题 1: 高阶 Folio 协同与写放大
          当 GC 处理一个高阶 folio（例如 64KB）时，它也必须与我们为 iomap 设计的
          f2fs_iomap_folio_state 机制协同工作。具体来说：

           • 私有标志位冲突: GC 过程可能会设置一些状态标志（如 PG_private_gcing）
                                                         ，
              这些标志需要被安全地存储在 f2fs_iomap_folio_state 中，以避免与 iomap
              自身的状态冲突。
           • 写放大风险: 如果 GC 只是因为 folio 中的一个 4KB 子页需要被回收，就
              将整个 64KB 的 folio 标记为脏并回写，这将导致严重的写放大。因此，GC
              路径也必须具备逐块脏页追踪的能力，精确地只回写那些真正需要被搬运的数
              据。

    问题 2: 因接口误用导致的双重解锁 (Double Unlock)
          这是一个更为隐蔽和危险的 bug，源于 F2FS GC 路径中一个令人困惑的接口使用
          惯例。在 GC 上下文中，fio->encrypted_page 字段的语义被复用，它实际上代表”
          任何需要后处理的页”，而不仅仅是加密页。GC 会将被选中的数据读到这个 page
          中，然后将其搬运到元数据 inode 对应的 meta_mapping 中。这个过程导致了灾难
          性的后果：
          Bug 复现路径：

           (a) 数据预读: GC 调用 ra_data_block 来读取需要被搬运的数据。
           (b) I/O 提交: 在 f2fs_submit_page_bio 中，由于 fio->encrypted_page 存

                在，bio 实际上被关联到了 meta_mapping 中的那个新 page，而不是原始的

3.7.3    解决方案

        为了解决上述问题，我们必须对 GC 的核心路径进行外科手术式的改造，使其具备真正的高阶
folio 意识。


        步骤 1: 实现精确的脏页标记
               我们在数据页的搬运函数 move_data_page 中，加入了对 folio 阶数（order）的
               判断。

                • 如果是 0 阶 folio，我们沿用旧逻辑，直接调用 folio_mark_dirty。
                • 如果是高阶 folio，则调用 iomap_set_range_dirty，它会利用 f2fs_iom
                 ap_folio_state 来精确标记出当前正在被 GC 的那个子页所对应的范围是脏
                 的，从而避免了对整个 folio 的不必要回写。

        步骤 2: 改造 I/O 提交逻辑，传递子范围信息
               这是解决双重解锁问题的关键。我们利用改造后的、支持范围的 f2fs_io_info 结
               构体，向 f2fs_submit_page_bio 传递了精确的块数量和页内偏移。
               通过这个改动，我们告诉块设备层，我们只关心 data_folio 中从 fio->idx 开始、
               长度为 fio->cnt 个块的区域。这样，即使 data_folio 是一个高阶 folio，I/O
               操作也只会针对其正确的子集进行，从而修复了之前错误的 I/O 范围。

        步骤 3: 同步改造 GC 的读路径
               GC 不仅涉及写，还涉及读。因此，我们也必须改造 GC 读取数据块的函数 f2fs
               _submit_page_read，使其在向 bio 添加页面时，也能正确传递子页在 folio 内
               的偏移。
               这一系列改动形成了一个完整的闭环，确保了 F2FS 的垃圾回收机制在读、写、脏
               页标记等所有关键路径上，都能正确、高效地处理高阶 folio，从根本上解决了写

        为了全面评估本项目中 f2fs 文件系统对 largefolios 支持所带来的性能影响，我们设计并
执行了一套严谨的性能测试方案。我们使用业界著名的存储性能测试工具 fio，针对普通文件、空
洞（sparse）文件以及压缩文件这三种典型场景，分别进行了缓存读（bufferedread）与缓存写
（bufferedwrite）的基准测试。
        测试平台覆盖了两种具有代表性的环境：标准的 x86-64 架构下的 QEMU 虚拟机，以及前沿的
ARM 架构裸金属设备——树莓派 5。特别地，在树莓派 5 上的测试基于最新的 16KB 页面大小配置进
行，其测试结果对于未来在 ARM 架构上推广 16KB 基页技术具有重要的参考价值和开创性意义。
        本报告将首先详细介绍我们的测试环境、核心测试脚本与方法论，随后展示在两大平台上的具
体性能测试结果并进行深入分析，最后介绍用于数据后处理与可视化的辅助脚本。


4.1       测试环境与方法论
        精准的性能测试依赖于一套自动化、可复现且能有效隔离干扰的测试流程。为此，我们编写了
一系列脚本来统一调度测试、设置环境、采集数据。

4.1.1    核心测试调度脚本：run_benchmark.py

        整个测试流程由核心调度脚本 run_benchmark.py 驱动。该脚本负责编排整个测试矩阵，包括
遍历不同的块大小（BlockSizes）
                   、读写模式以及多队列深度（QueueDepth）
                                          ，并为每组测试配置执
行指定次数的迭代，以获取稳定、可靠的性能数据。
4   项目性能测试                                                                    85


     该脚本的核心职责包括：

    • 参数解析与矩阵生成：解析用户输入（如内核版本标识、测试类型、目标文件等）
                                         ，并与预定义
     的块大小列表（4k,128k,1M,10M,100M,1G）组合，生成完整的测试任务队列。

    • 环境“写栅栏”设置：在进行任何读写测试之前，调用 set_write_fence() 函数。此函数的
     目的是为了最大限度地减少内核后台任务对测试结果的干扰。在标准的 Linux 内核中，当应
     用程序向页缓存（PageCache）写入数据后，这些“脏页”会由后台的回写（writeback）机
     制在稍后的某个时间点异步地刷写到磁盘。这种异步回写行为会占用 CPU 和 I/O 带宽，从而
     对我们测量的纯缓存读写性能造成干扰。set_write_fence() 通过修改 sysctl 内核参数（如
                                                                            ，临时性
     vm.dirty_ratio, vm.dirty_background_ratio, vm.dirty_writeback_centisecs）
     地“关闭”这个自动回写机制，确保在测试期间所有写操作都停留在内存中，从而精确测量缓
     存性能。测试结束后，restore_write_fence() 函数会负责将这些参数恢复原状。

    • 只读测试环境隔离：在执行“冷读”测试前，脚本会调用 remount() 函数，将目标文件系统以
     只读（read-only）模式重新挂载。这样做是为了彻底杜绝在清理缓存（调用 sync 命令）时
     可能引发的元数据更新（如访问时间 atime），确保读测试的纯粹性。

    • 迭代调用 fio：对测试矩阵中的每一个测试点，脚本会循环调用底层的 fio_test.sh 脚本执
     行 10 次，并将结果分别记录。这种多次迭代取平均值的做法，旨在平滑由 CPU 调度抖动或其
     他瞬时系统活动带来的数据波动，使最终结果更具统计意义。

    ∗ BW/bw、IOPS 大小写不敏感
    ∗ IOPS 支持 k / m 后缀并换算为「次/秒」

    import re, csv, math
    from pathlib import Path


    RESULTS_DIR = ”./fio_results”
    OUTPUT_CSV     = ”./results.csv”


    # −−−−−−−−−− 小工具 −−−−−−−−−− #


    def _parse_with_suffix(value: str) −> float:
          ”””把 '212k' / '15.3m' / '987' 统一转成 float”””
          if value.lower().endswith('k'):
                return float(value[:−1]) ∗ 1_000
          if value.lower().endswith('m'):
                return float(value[:−1]) ∗ 1_000_000
          return float(value)


    # −−−−−−−−−− 解析单个 log −−−−−−−−−− #


    def parse_fio_log(path: Path):
          txt      = path.read_text()
          fname    = path.name


          # 1) 文件名元数据 −−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−
          m = re.match(
          r”^(?P<kernel>[\w\−]+)_(?P<mem>\d+G)_mem_”
          r”(?P<mode>read|write|sync)_bs−(?P<bs>[\w\d]+)_(?P<ts>[\d\−]+)\.log$”,
          fname)
          if not m:
                print(f”[skip] 文件名不匹配: {fname}”)
                return None
          meta = m.groupdict()
4   项目性能测试                                                                  104



      # 2) 性能指标 −−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−
      # BW     / IOPS
      bw_rg      = re.search(r'BW[=|:](\d+\.?\d∗)[GM]iB/s', txt, re.I)
      iops_rg = re.search(r'IOPS=(\d+\.?\d∗[kKmM]?)', txt, re.I)


      # clat 行（可选）
      clat_rg = re.search(r'clat \((usec|msec)\):.∗?avg= ∗([\d.]+)', txt)
      p99_rg    = re.search(r'99\.00th=\[\s∗([\d.]+)]', txt)

      cpu_rg    = re.search(
             r'^.∗cpu.∗usr=([\d.]+)%.∗sys=([\d.]+)%', txt,
             flags=re.I | re.M)


      # −−−−−−−− 校验 & 容错 −−−−−−−−−− #
      if not (bw_rg and iops_rg and cpu_rg):

      bw = float(bw_rg.group(1))
      if ”GiB/s” in bw_rg.group(0):
             bw ∗= 1024             # 统一成 MiB/s


      iops = _parse_with_suffix(iops_rg.group(1))    # 转成 “次/秒”

             if ”clat percentiles (msec)” in txt:
                 clat_p99 ∗= 1000


      cpu_usr = float(cpu_rg.group(1))
      cpu_sys = float(cpu_rg.group(2))


      return {∗∗meta,
                 ”bw_mib_s”:        bw,

5.1.2   高级工作流与遇到的问题


     问题 1: 多版本内核并行开发与异地编译
            为了进行性能基准测试，我需要同时维护和编译多个不同版本的内核（如主线开发
            版、我的补丁版、用于对比的稳定版）。为保持源码树的干净，我采用了异地编译
           （Out-of-Tree Build）的策略。

     问题 2: 复杂的多实例 QEMU 管理
            开发过程中，经常需要一边运行带调试符号的内核进行 GDB 调试，另一边运行优化
            后的内核进行性能测试。手动管理多个 QEMU 实例的命令行参数、镜像和 PID 非
            常繁琐。为此，我编写了一个强大的多实例管理脚本。
            核心功能解析：

             • 实例隔离: 为每个实例（如 vm1, vm2）创建独立的目录，存放其差分镜像、共
                 享目录和 PID 文件，互不干扰。
             • 写时复制 (CoW) 镜像: 使用 qemu-img 基于一个只读的基准系统镜像创建
                 qcow2 差分镜像。这极大地节省了磁盘空间，并能快速创建和销毁实例。
             • 灵活配置: 通过命令行参数可以方便地为不同实例指定不同的内核目录 (-k)
                 和内存大小 (-m)。
             • 生命周期管理: 提供了 start, stop, cleanup 三个命令，实现了对虚拟机实例
                 从创建、运行到彻底销毁的全生命周期管理。
             • 主机-虚拟机文件共享: 利用 -virtfs 参数实现了两个共享目录，一个用于通
                 用文件交换，另一个专门用于动态加载和卸载内核模块 (.ko 文件)，极大地方
                 便了调试。

     问题 3: 虚拟机内 F2FS 自动挂载
            每次启动虚拟机后手动挂载 F2FS 分区并加载模块非常低效。为了实现自动化，我
            编写了一个 systemd 服务，在开机后自动完成所有准备工作。
            服务单元解析：

     • 交互式变基 (gitrebase-i): 这是最重要的工具。我用它来：

         – 拆分 Commit: 将一个大的功能实现（如 buffered read/write）拆分成多个逻辑独
           立的、原子性的提交。
         – 合并 Commit (squash): 将一些小的修复或调试性提交合并到其所属的功能提交

         – 编辑 Commit (edit): 修改提交信息或对某个历史提交进行微调。

        结果具有极高的现实参考价值和前瞻性。




    部署流程

    1. 格式化 16KB 块大小的 F2FS 分区: 树莓派官方 Ubuntu 22.04 源中的 f2fs-tools 版本
        较旧，不支持格式化 16KB 块大小。为此，我从官方 Git 仓库拉取了最新的 f2fs-tools
        源码，在我的 x86 主机上交叉编译出 aarch64 版本的二进制文件。然后，我将树莓派的
        SSD 通过 USB 转接器挂载到主机上，使用这个新编译的 mkfs.f2fs 成功格式化出了
        16KB 块大小的 F2FS 分区。

    2. 内核编译与部署策略: 为了确保系统的稳定性和驱动兼容性，我采取了稳健的策略：

        (a) Fork 树莓派官方的内核源码仓库（版本 6.15）。
5   开发环境搭建与工作流                                                                     119



       (b) 首先，不加任何修改地编译官方内核，并将其部署到树莓派上，确保可以成功启动。
        (c) 验证成功后，在我的 Fork 中开辟新分支，只修改文件系统（fs/f2fs/）相关的代
           码，不触碰任何驱动或核心架构代码。

       这个策略保证了我的内核能够基于一套官方验证过的稳定驱动运行，极大地降低了调试
       难度。

     3. 自动化部署脚本: 手动复制内核镜像、设备树和模块到 SD 卡或通过网络传输非常繁琐
       且容易出错。为此，我编写了一个全自动的部署脚本。

    # Deploy a custom RPi5 kernel to /boot/firmware and modules to /usr/lib/
       modules (on the Pi)
    # Requirements on host: built kernel tree, sshpass, scp/ssh
    set −Eeuo pipefail


    ### ==== 配置区（按需修改） ====
    # 内核源码目录（已完成配置并能正常 make 的目录）
    KERNEL_SRC=”${KERNEL_SRC:−$PWD}”


    # 交叉编译前缀（如果本机是 x86_64 而编 aarch64 内核）
    CROSS_COMPILE=”${CROSS_COMPILE:−aarch64−linux−gnu−}”
    ARCH=”${ARCH:−arm64}”


    # 树莓派 5（BCM2712）要放到 /boot/firmware 的镜像名
    KERNEL_IMAGE_NAME=”${KERNEL_IMAGE_NAME:−mykernel_2712.img}”


    # 目标树莓派
    RPI_HOST=”${RPI_HOST:−raspberrypi.local}”
    RPI_USER=”${RPI_USER:−pi}”
    RPI_PASS=”${RPI_PASS:−pi}”
    RPI_BOOT=”${RPI_BOOT:−/boot/firmware}”       # 你的系统就是 /boot/firmware
    RPI_MODBASE=”${RPI_MODBASE:−/usr/lib/modules}”# Ubuntu/RPiOS 下 /lib/modules
        −> /usr/lib/modules


    # 本地打包输出目录（安全区，不触碰系统目录）
    STAGING=”${STAGING:−/tmp/rpi−kernel−out}”

    command −v sshpass >/dev/null || { echo ”需要安装 sshpass”; exit 1; }

    make −C ”$KERNEL_SRC” ARCH=”$ARCH” CROSS_COMPILE=”$CROSS_COMPILE” −j”$(nproc
       )” Image modules dtbs

    KREL=”$(make −sC ”$KERNEL_SRC” ARCH=”$ARCH” CROSS_COMPILE=”$CROSS_COMPILE”

    echo '部署完成。新内核: '\$KREL'            镜像: '\$RPI_BOOT'/$KERNEL_IMAGE_NAME'
    ”

5.3.2   遇到的重大问题：动态 IP 与网络穿透

     问题的根源：不稳定的局域网 IP
         在校园网环境中，树莓派通过 Wi-Fi 或有线连接获取到的 IP 地址是动态分配的，每次
     重启或重连后都可能改变。这使得我无法在 WSL 开发环境中通过一个固定的 IP 地址来 SSH
     连接或使用 SCP 传输文件，极大地阻碍了开发流程。




         失败的尝试
         我曾尝试使用 Windows 系统自带的移动热点功能，
                                   让树莓派连接此热点以获得固定 IP。
     然而，该热点创建的网络与我的 WSL 所在的 Hyper-V 虚拟网络处于不同的局域网段。尽管
     我手动配置了 WSL 的路由表试图进行转发，但在排除了防火墙问题后，两者之间依然无法
     ping 通。我推测这可能是 Hyper-V 虚拟交换机本身的限制所致。




         最终解决方案：使用 WireGuard 搭建虚拟私有网络
         为了彻底解决这个问题，
                   我租用了一台具有公网 IP 的云服务器
                                     （VPS）
                                         ，并使用 WireGuard
     在我的 WSL、VPS 和树莓派之间构建了一个星形拓扑的虚拟私有网络（VPN）。
         工作原理:

        • VPS 作为中心节点（中转服务器）
                          ，它拥有一个固定的公网 IP 和一个固定的 VPN 内网
          IP (如 10.0.0.1)。

        • 我的 WSL 和树莓派都作为客户端连接到这个中心节点。它们各自也被分配一个固定的
          VPN 内网 IP (如 WSL 为 10.0.0.2，树莓派为 10.0.0.3)。

        • 流量通过 WireGuard 的加密隧道，经由 VPS 进行中转。这样，无论我的 WSL 和树莓
          派在物理世界中的 IP 地址如何变化，它们在 VPN 这个” 虚拟局域网” 中的 IP 地址永
          远是固定的。

         配置流程:

        1. 在三台设备上都安装 WireGuard

        2. 生成密钥: 在每台设备上都生成一对公钥和私钥。

        3. 配置 VPS (服务器):
