 # 前四章架构整理

本文前四章不采用传统“技术综述-系统设计-系统实现”的平铺叙事，而采用“问题导向的分层递进叙事”。其总逻辑如下：

1. 第一章先提出问题，明确为什么值得研究。
2. 第二章补齐理解问题所必需的层级关系与背景知识。
3. 第三章把第一章总说的问题拆成真正的技术问题。
4. 第四章再逐项回应第三章的问题，给出对应设计与实现。

换言之，前四章不是四块松散内容，而是一条连续的问题链。

## 第一章架构点

### 第一章的职责

第一章不是技术细节章，也不是实现概览章。第一章只做三件事：

1. 说明为什么当前移动终端和端侧 AI 场景需要重新审视文件系统 I/O 路径。
2. 说明本文的问题链是什么，而且是分层递进的，不是一锅端的大概念。
3. 说明本文为什么选择 F2FS、Large Folio 和 iomap 作为研究对象与技术路线。

### 第一章的问题链

第一章必须明确区分两层问题。

#### 第一层问题

传统 4KB 页缓存粒度过小，难以充分发挥现代块层和闪存设备的大块连续 I/O 能力。

这一层问题对应的技术意义是：

- Large Folio 的引入首先是为了扩大页缓存对象粒度；
- 它先解决“页缓存对象太小”的问题；
- 此时不必急于把它定义成全部意义上的“粒度失配”。

#### 第二层问题

即使引入 Large Folio，在 F2FS 中仍然存在更深层的不一致：更大的 folio 会覆盖多个文件系统块、多个子区间、多个 bio 或多个完成回调，因此映射边界、状态边界以及 I/O 提交与完成生命周期之间难以自然对齐。

这一层问题对应的技术意义是：

- iomap 的引入主要是为了回答这一层问题；
- 这里再正式引入“粒度失配”或“边界失配/生命周期失配”更准确；
- 这样才能把 Large Folio 和 iomap 的意义分开。

### 第一章必须保留的内容

1. 端侧 AI、大文件模型加载、移动应用持续增长的 I/O 背景。
2. Android 常用 F2FS 的场景定位。
3. 4KB 页缓存粒度过小的问题。
4. Large Folio 作为第一步技术的意义。
5. F2FS 中仍然存在第二层问题。
6. iomap 作为第二步技术的意义。
7. 本文工作和章节安排。

### 第一章只能点到为止的内容

第一章可以提 F2FS 更复杂，但只能作为“问题放大器”出现，不能展开成完整背景章。

第一章只需点出：

- F2FS 采用日志结构；
- 存在异地更新、GC、压缩文件等机制；
- 因此 Large Folio 引入后问题不只是普通读写性能问题，还会传导到写回、搬移和压缩路径。

### 第一章的叙事关键词

建议固定使用：

### 第二章的职责

2. 再用一个走读例子把它们串起来（2.5）

4. 最后小结（2.9）

### 第二章的章节骨架

#### 2.1 页缓存与页描述符

讲清页缓存的基本角色和核心数据结构。重点：
- 页缓存位于 VFS 之下、文件系统之上，以页描述符管理被缓存的文件数据
- `struct page` 的核心字段：`index`（文件内的页索引 = 文件偏移 / 4096）、`mapping`（指向所属 inode 的 address_space）、`flags`（PG_uptodate/PG_dirty/PG_locked 等状态位）
- 缺页流程：页缓存查找 → 未命中 → 分配页描述符 → 加锁 → 请求文件系统填充数据 → 解锁 → 标记 uptodate
- 不讲 xarray 存储细节

#### 2.2 F2FS 块映射

讲清文件系统如何回答"这块数据在磁盘的什么位置"。重点：
- 文件偏移 → 逻辑块号（偏移 / 块大小）→ 物理块号（通过 NAT 表查询）
- NAT 的作用：维护节点块（inode、直接/间接节点块）的逻辑地址到物理地址的映射
- `f2fs_map_blocks` 接口：输入逻辑块号，返回物理块号、连续长度和区间状态（mapped/hole/new）

#### 2.3 bio 与多页 I/O 提交

讲清块层如何组织并提交 I/O。重点：
- bio 是块层提交 I/O 的基本载体，内部通过 `bio_vec` 数组描述多段物理连续的内存区间
- `bio_add_page` 将页的物理内存段加入 bio，物理连续时合并到同一 bvec，不连续时新增 bvec
- `bi_sector` 表示目标设备的起始扇区（512 字节），由物理块号换算得来
- bio 提交后由块设备驱动执行 DMA，完成后触发 `bi_end_io` 回调
- 关键结论：一个 bio 可以承载多个页的数据——块层不要求每次 I/O 只能处理一个 4KB 页

#### 2.4 闪存逻辑地址空间与 FTL

讲清闪存设备的物理约束和 FTL 的角色。重点：
- NAND 闪存的不对称性：写入以页（4KB-16KB）为单位，擦除以块（数百 KB-数 MB）为单位；一个页在擦除前只能写入一次
- FTL 在固件层隐藏这种复杂性，向上呈现普通逻辑块设备；FTL 维护 LBA→物理闪存页的映射表，执行磨损均衡和 GC
- 原地覆盖的代价：改写一个 4KB 页，FTL 可能需执行"读整块→擦整块→重写"的读改写链 → 写放大
- 闪存友好的策略是顺序追加写入而非随机原地覆盖

#### 2.5 走读例子：一次 64KB buffered read 的完整路径

将前四节串联成一个自上而下的走读。用 `read(fd, buf, 64KB, file_offset=0)` 逐步展开：
1. 系统调用进入 VFS，转入 `generic_file_read_iter`
2. 页缓存按文件偏移计算页索引，在 inode 的 address_space 中查找
3. 未命中 → 分配页描述符 → 加锁 → 调用 `f2fs_readahead` / `f2fs_map_blocks`
4. F2FS 将逻辑块号转换为物理块号并返回连续长度
5. 块层根据物理块号换算 `bi_sector`，调用 `bio_add_page` 将多个页的物理段加入 bio，`submit_bio` 提交
6. 块设备执行 DMA 读取，bio 完成后回调 `iomap_read_end_io`，标记页 uptodate，解锁
7. 数据拷贝回用户缓冲区
讲解粒度：函数调用名可以列出，但不贴源码

#### 2.6 Large Folio：可变阶数动态大页

在走读之后自然引出：刚才 64KB 走了十六遍分配→加锁→映射→提交，能否一遍？讲清：
- `struct folio` 与 `struct page` 的关系：folio 复用头页的 page 结构，但类型上保证绝不指向 tail page
- order 的含义：order=0 为 4KB，order=2 为 16KB，order=4 为 64KB
- Large Folio 就是 order ≥ 1 的 folio，一个对象覆盖更大连续文件区间
- 页缓存通过 readahead 窗口和访问模式动态决定 order
- 直接效果：分配次数、锁操作、映射查询次数随对象数量下降

#### 2.7 F2FS 的异地更新、GC 与压缩文件

基于 2.4 的闪存约束，讲清 F2FS 为了适应闪存而引入的机制：
- 异地更新：数据更新时新版本写入新物理段，旧块标记无效——直接回应 2.4 的"不能原地覆盖"
- F2FS 关键元数据区：NAT（节点地址映射）、SIT（段利用率）、SSA（段摘要）、Checkpoint、Main Area——只讲"各自管什么"
- GC：无效块积累 → 选择 victim 段 → 识别有效块 → 搬移到新段 → 擦除旧段。GC 会重新进入页缓存路径
- 压缩文件：以 cluster（通常 2-4 个块）为单位压缩/解压，cluster 边界不与 block 边界一致

#### 2.8 iomap：字节区间映射

基于 2.2 的逐块映射和 2.6 的 Large Folio，讲清 iomap 的核心设计：
- buffer_head 是逐块状态对象，和 Large Folio 的大粒度矛盾
- iomap 以字节区间[offset, offset+length)描述映射，一次返回覆盖连续区间的映射结果
- iomap_iter 迭代器模式：文件系统实现 begin/end 回调，框架自动循环消费区间
- 同一套 iomap_ops 同时服务 buffered read、buffered write、writeback——统一 I/O 组织
- 和 Large Folio 的协同：一个物理连续的 64KB folio，一次 iomap_begin 返回一个覆盖全区的映射
- 不讲：iomap_folio_state 内部位图布局（留给第四章）、F2FS 回调的具体实现

#### 2.9 本章小结

收束：2.1-2.4 建立四层概念基础，2.5 用一个例子贯穿，2.6-2.8 依次引出本文的三层技术基础——Large Folio 解决页缓存粒度过小（第一层），iomap 提供字节区间 I/O 组织（第二层），F2FS 的异地更新/GC/压缩机制揭示了第三层的兼容挑战。

### 第二章的核心收束句

"通过上述背景知识可以看出：从页缓存到闪存的各层对'数据范围'的理解天然不同——页描述符讲 4KB 缓存对象，F2FS 讲逻辑块映射，bio 讲物理连续区间。Large Folio 通过扩大页缓存对象粒度回应第一层问题；iomap 通过字节区间映射将文件系统 I/O 从逐块模式提升为区间模式，回应第二层问题；而 F2FS 的异地更新、GC 和压缩机制则在 iomap 的通用框架之上要求进一步的专属适配，构成了第三章将要分析的核心困难。"

## 第三章架构点

### 第三章的职责

第三章不再讲背景知识，也不再讲解决方案。第三章只做一件事：把第二章已建立的各层机制放到 F2FS 的具体路径中，指出它们在 Large Folio 条件下哪里会出现冲突、为什么会出现冲突。为第四章的设计与实现提供一一对应的问题来源。

### 第三章的内部分节

#### 3.1 本章定位

明确第三章边界：不讲背景知识，不讲解决方案。只将 F2FS 已有路径放到 Large Folio 条件下检验，指出冲突在哪、为什么冲突。

#### 3.2 F2FS 所有路径的公共前提：一个页缓存对象=一个块

从 F2FS 源码出发，说明四条路径（读、写准备、写回、GC 搬移）中"一个 folio 等于一个文件系统块"这一隐含假设的具体形态。这是全章问题分析的共同起点。

#### 3.3 私有字段的所有权冲突（Layer 3 状态冲突）

分析 `folio->private` 在 F2FS（直接编码 GC/原子写标志）和 iomap（保存 per-folio 位图指针）两条路径中的双重解释，展示实际崩溃路径，说明 order-0 folio 同样受影响。

#### 3.4 压缩文件的簇边界与 Large Folio 边界的不重合（Layer 3 压缩问题）

分析压缩簇（2~4 块）与 Large Folio 边界不重合：一个 folio 可能跨越多个簇，不同类型子范围需要不同填充策略，混合 I/O 模型导致完成时机难以统一。

#### 3.5 脏页写回中的写放大与子范围控制（跨 Layer 2+3 写回问题）

分析传统写回路径将 folio 拆为 page 数组后的三个后果：脏区信息粗化、写放大（局部脏→整 folio 追加写入→日志消耗→GC 压力增加）、同一对象被重复加锁和提交。

### 第三章与第四章的对应关系

- 3.3（私有字段冲突）→ 4.3（统一状态结构设计）
- 3.4（压缩文件）→ 4.6（压缩文件扩展处理）
- 3.5（写回控制）→ 4.4（脏页写回粒度控制）

### 第三章的核心收束句

"第三章的分析表明，F2FS 支持 Large Folio 的核心困难并不在于分配更大的缓存对象本身，而在于全路径中默认的'一个对象=一个块'前提被打破后，映射、状态和完成三个维度在 F2FS 特有机制（异地更新、GC、压缩）的放大下如何系统性地失去对齐。这些问题的具体回应将在第四章中给出。"
