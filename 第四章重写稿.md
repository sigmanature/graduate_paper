# 第四章 F2FS Large Folio 缓冲 I/O 设计与实现

## 4.1 总体架构

本章以第三章的问题链为输入，给出 F2FS Large Folio 缓冲 I/O 的整体设计与实现方案。图 4-1 展示了本系统实现的总体架构。

架构的顶层是 iomap 框架中被 F2FS 复用的五个组件：`iomap_readahead`（①，缓冲读）、`iomap_file_buffered_write`（②，缓冲写）、`writeback_iter`（③，脏页写回迭代器）、`iomap_iter`（④，迭代器数据结构）和 `iomap_readpage_iter`（⑤，通用读迭代器）。F2FS 不改动这些组件本身，仅通过实现对应的 `iomap_begin` 回调嵌入自身逻辑。三条回调路径在内部统一调用 `f2fs_set_iomap`（⑥，4.2.1 节）将 F2FS 的块级映射结果转换为 iomap 字节区间描述符，并通过 `f2fs_folio_state`（⑦，4.3 节）维护 per-block 状态位图和完成计数（⑧）。这八项机制构成了被所有路径复用的基础。

在核心机制之上，GC 路径和压缩文件路径叠加各自的专属逻辑。GC 适配（4.6 节）复用 ⑥⑦⑧ 并扩展子范围偏移修正。压缩文件处理拆为两条独立路径：压缩读（4.4.1 节）复用 ④（`iomap_iter`）并以自实现的 `f2fs_compress_readpage_iter` 复刻 ⑤（`iomap_readpage_iter`），同时扩展多簇物理块信息一次性获取能力；压缩写（4.4.2 节）复用 ②（`iomap_file_buffered_write`）的 `f2fs_buffered_write_iomap_begin`，并扩展跳过完全覆盖中间簇的优化。所有路径最终经 bio 层统一提交到块设备。

## 4.2 普通文件区间映射 I/O

### 4.2.1 块映射到 iomap 区间的转换：f2fs_set_iomap

F2FS 的块映射接口 `f2fs_map_blocks` 以逻辑块号 `m_lblk`、物理块号 `m_pblk` 和连续块数 `m_len` 为单位返回映射结果，而 iomap 框架以文件字节偏移 `offset`、设备字节地址 `addr` 和字节长度 `length` 描述映射区间。当 Large Folio 在 I/O 路径中的覆盖面从 1 块扩展到 16 块后，原先隐藏在逐块调用中的语义分歧集中暴露——各路径对 `f2fs_map_blocks` 返回结果的解释方式不统一，空洞边界、新分配区间和已映射块的类型判定逻辑分散在多处调用点中，任何一处不一致都会导致区间类型误判。第三章 3.2 节展示的全路径单块假设，其直接技术表现正是这种分散的逐块转换。

为此，本系统将块映射到字节区间的转换逻辑收敛到单一函数 `f2fs_set_iomap` 中。该函数同时服务于 buffered read、buffered write 和 direct I/O 三条路径，其核心逻辑如算法 4-1 所示。

{{ALG:f2fs_set_iomap}}

该函数在三种映射状态间做出精确区分。当 `F2FS_MAP_MAPPED` 已置位时，进一步检查 `m_pblk`：若为有效物理块号则输出 `IOMAP_MAPPED` 类型并设置 `IOMAP_F_MERGED` 标志（表示该区间来自 extent cache 合并后的连续结果）；若为 `NEW_ADDR` 则输出 `IOMAP_UNWRITTEN`（延迟分配尚未落盘）。当映射未命中时，`m_pblk == NULL_ADDR` 表示空洞，此时通过 `F2FS_MAP_NODNODE` 区分两种空洞：若该位置无直接节点条目，空洞终点以 `m_next_pgofs` 为界（即下一个存在节点块的逻辑页号）；若有节点条目但映射为空，则以 `m_len` 为界。最后，`F2FS_MAP_NEW` 和 inode 脏状态分别被转换为 `IOMAP_F_NEW` 和 `IOMAP_F_DIRTY` 标志。

引入该转换层之前，F2FS 原生读接口每次只请求单个块的映射（`map.m_len = 1`）。对一个 64KB（16 块）的 Large Folio，这意味着 16 次 `f2fs_map_blocks` 调用和 16 次独立的块映射结果解释。转换层建立后，上层的 `iomap_begin` 回调将整个区间请求一次性传递给 `f2fs_map_blocks`，后者内部的 `map_is_mergeable` 逻辑自动合并物理和逻辑连续的相邻块，使 `map.m_len` 扩大到覆盖整个连续区间。`f2fs_set_iomap` 随后将这一扩展后的结果一次性转换为 iomap 字节区间。在 extent cache 命中的理想情况下，映射调用从 16 次降至 1 次，锁操作次数同步下降。

### 4.2.2 缓冲读路径

第三章 3.2 节描述了原生单页读函数 `f2fs_read_single_page` 的逐块工作方式：每次调用仅向 `f2fs_map_blocks` 请求一个逻辑块的映射，然后将其单独加入 bio。当 Large Folio 将单次覆盖范围从 1 块扩展到 16 块后，这一路径的映射调用次数、锁操作次数和 bio 提交次数同步放大 16 倍——逐块循环本身成了瓶颈。

为此，本系统用 `f2fs_buffered_read_iomap_begin` 回调（算法 4-2）将上述逐块循环替换为单次区间映射请求。

{{ALG:f2fs_read_iomap_begin}}

该回调的核心操作是两次转换：先将字节区间 `(offset, length)` 转换为逻辑块号区间 `(map.m_lblk, map.m_len)`，在 `f2fs_map_blocks` 返回后通过 4.2.1 节的 `f2fs_set_iomap` 将块映射结果转换为 iomap 区间描述符。其中的关键标志是 `F2FS_GET_BLOCK_IOMAP`——它告知 `f2fs_map_blocks` 内部遍历循环在块映射信息足以构建 iomap 时立即返回，而非遍历到直接节点页末尾，从而将返回结果的粒度控制在"连续映射子区间"水平。

`f2fs_map_blocks` 内部为读路径提供两条查找路径。extent cache 快速路径在 `extent_info` 条目命中时直接从缓存计算 `m_pblk = ei.blk + (pgoff - ei.fofs)` 和合并后的 `m_len`，无需产生磁盘 I/O。dnode 常规路径在 cache 未命中时逐块遍历，调用 `map_is_mergeable` 合并物理和逻辑均连续的相邻块——`(map->m_pblk + map->m_len == blkaddr) && (map->m_next_extent == pgofs)`——每合并一个块，`m_len` 递增 1。遍历结束后将合并后的连续区间写入 extent cache。

在 `f2fs_set_iomap` 填充好 iomap 之后，剩余工作由 iomap 框架的 `iomap_iter` 迭代器自动完成：它根据 iomap 长度推进迭代位置，每次迭代调用 `iomap_readpage_iter` 将当前区间内的页面加入 bio、管理 uptodate 位图，并在整个 folio 的所有子区间全部就绪后解锁。文件系统不需要关心 bio 构建、完成回调注册或解锁时机的具体实现。

表 4-1 对比了原生路径与本系统路径在 64KB Large Folio 场景下的关键指标。

| 指标 | 原生路径（f2fs_read_single_page） | 本系统路径（f2fs_buffered_read_iomap_begin） |
|------|-----------------------------------|------------------------------------------|
| `f2fs_map_blocks` 调用次数 | 16 次（逐块） | 1 次（extent cache 命中）或 N 次（N=连续子区间数） |
| extent cache 利用 | 逐块重复查找 | 一次大区间返回合并后结果 |
| 块合并 | 不适用（每次只请求 1 块） | `map_is_mergeable` 遍历中自动合并 |
| dnode 锁操作 | 16 次 | 1 次 |
| bio 提交 | 16 次独立提交 | iomap 框架自动合并为更少提交 |

### 4.2.3 缓冲写路径

F2FS 原生写入准备函数 `prepare_write_begin` 围绕单个逻辑块工作，一次只分配一个空闲块（`f2fs_reserve_new_block`），所有元数据操作——dnode 查找、段锁获取、空间预留——均为逐块执行。当 Large Folio 使单次写入的块数从 1 上升到 16 后，这种"分配一个、写入一个、再分配下一个"的循环使元数据操作开销与块数线性增长，且底层块合并能力（`map_is_mergeable`）因上层每次只请求 1 块而完全闲置。

针对这一瓶颈，本系统用 `f2fs_buffered_write_iomap_begin` 回调（算法 4-3）将逐块分配替换为区间级预分配。

{{ALG:f2fs_write_iomap_begin}}

该回调分两个阶段。第一阶段以 `F2FS_GET_BLOCK_IOMAP` 模式调用 `f2fs_map_blocks_iomap`，查询目标写入区间内是否已有物理块映射。第二阶段仅在区间为空洞时触发：调用 `f2fs_map_blocks_preallocate`，内部以 `F2FS_GET_BLOCK_PRE_AIO` 标志再次驱动 `f2fs_map_blocks`，在遍历到直接节点页末尾时调用 `f2fs_reserve_new_blocks` 进行批量空间预留——一次调用为整个写入区间分配连续空闲块，返回一个横跨多个逻辑块的、以 `NEW_ADDR` 为物理地址的统一映射。

批量预分配能生效的关键在于 `map_is_mergeable` 对 `NEW_ADDR` 的特殊处理：当当前块和前一映射都是 `NEW_ADDR` 时，合并条件同样成立——`(map->m_pblk == blkaddr && blkaddr == NEW_ADDR)`。这使连续的新分配区间被合并为一个大区间返回，而非退化为每次一个块的细粒度分配。

在 iomap 框架层面，写路径享有两项额外收益。folio 生命周期管理由框架自动完成：`iomap_write_begin` 负责查找或创建 folio 并加锁，文件系统回调在安全的已锁定上下文中执行，无需手动处理并发截断竞争。脏区标记精度也得到提升：`__iomap_write_end` 通过 `iomap_set_range_dirty(folio, offset_in_folio, copied)` 将 folio 内被实际修改的字节区间标记为脏，而非整 folio 标记——这对后续 4.5 节的子范围写回控制至关重要。

表 4-2 对比了原生逐块写准备路径与本系统区间预分配路径。

| 指标 | 原生路径（prepare_write_begin） | 本系统路径（f2fs_buffered_write_iomap_begin） |
|------|-------------------------------|-------------------------------------------|
| 空间分配 | 逐块分配，每次 1 个 NEW_ADDR | 区间预分配，一次 N 个 NEW_ADDR |
| dnode 查找 | N 次 | 1 次 |
| `f2fs_reserve_new_block` | N 次 | 1 次（内部批量） |
| 连续块合并 | 每次 1 块，无法合并 | `map_is_mergeable` 合并连续 NEW_ADDR |
| 原子文件 | 独立复制 dnode 逻辑 | 复用 `f2fs_map_blocks_iomap`，仅切换 inode |

## 4.3 统一状态结构：f2fs_folio_state

第三章 3.3 节揭示了 `folio->private` 字段的所有权冲突：F2FS 在其中直接编码 `ONGOING_MIGRATION` 等整数标志，而 iomap 框架期望同一字段保存指向 `iomap_folio_state` 的指针。两条路径对该字段赋予互不相容的解释——一方做位测试，一方做指针解引用——当同一 folio 先后进入 GC 路径和 iomap 读写路径时，读完成回调以地址 0x4 附近的"指针"解引用，直接触发内核 panic。0 阶 folio 同样存在此风险：GC 路径向 0 阶 folio 写入 `ONGOING_MIGRATION` 标志后，该 folio 完全可能随后进入 iomap 缓冲写路径。

针对这一字段级别的语义冲突，本系统设计了一个与 `iomap_folio_state` 头部内存兼容、但通过柔性数组扩展额外承载 F2FS 私有标志和压缩 pending 计数的统一状态结构 `f2fs_folio_state`。

```
struct f2fs_folio_state {
    spinlock_t  state_lock;
    unsigned int read_bytes_pending;
    atomic_t    write_bytes_pending;
    unsigned long state[];  // 柔性数组
};
```

前三个成员（`state_lock`、`read_bytes_pending`、`write_bytes_pending`）与 `iomap_folio_state` 的定义完全一致，确保内存布局兼容。例如，iomap 的 `iomap_finish_folio_read` 通过 `(struct iomap_folio_state *)folio->private` 访问 `read_bytes_pending` 字段时，其偏移量与 `f2fs_folio_state` 中同一成员的偏移量相同——操作完全等价。

柔性数组 `state[]` 在 `iomap_folio_state` 所需字节之后多分配两个 `unsigned long` 的空间。第一个额外空间（`state[iomap_longs]`）存放 F2FS 的所有私有标志位（`ONGOING_MIGRATION`、`ATOMIC_WRITE`、`DEFERRED_UNLOCK` 等），取代原 `folio->private` 低位直接编码标志的方式。第二个额外空间（`state[iomap_longs+1]`）作为 `dirty_bytes_pending` 原子计数器，供压缩文件写回路径的延迟解锁使用（4.5.2 节）。

由于 `folio->private` 可能指向 `iomap_folio_state` 或 `f2fs_folio_state`，运行时必须区分两者。本系统在分配 `f2fs_folio_state` 时，将魔数 `F2FS_IFS_MAGIC` 写入 `read_bytes_pending` 字段——该字段在 iomap 框架中仅在读 I/O 进行时才携带真实计数值，初始化后处于空闲状态。类型检查函数 `is_f2fs_ifs` 通过读取该字段并与魔数比对完成识别。

结构体的核心分配函数 `f2fs_ifs_alloc`（算法 4-4）处理三种边界情况。

{{ALG:f2fs_ifs_alloc}}

第一种，`folio->private` 为空：直接分配并初始化新的 `f2fs_folio_state`，根据 folio 当前 uptodate/dirty 状态初始化对应位图。第二种，`folio->private` 已指向 `iomap_folio_state`（魔数不匹配）：分配更大的 `f2fs_folio_state`，将旧的逐块位图内容通过 `ifs_to_f2fs_ifs` 拷贝到新结构体对应位置，写入魔数，通过 `folio_change_private` 原子替换指针并释放旧结构体。第三种，魔数匹配：直接返回已有指针。这一分配-拷贝-替换的迁移路径保证了 iomap 已有的逐块状态在 F2FS "接管" `folio->private` 时不丢失。

为兼容 F2FS 原有的大量 `set_page_private_gcing`、`folio_test_private_atomic` 等调用点，本系统设计了新的宏族 `F2FS_FOLIO_PRIVATE_GET/SET/CLEAR_FUNC`。以 Get 操作为例：首先检查 `folio->private` 是否已设置 `PAGE_PRIVATE_NOT_POINTER` 标志——若是，说明存储的是旧式直接标志位，直接在该值上进行位操作（快速路径，兼容元数据 folio）；若否，说明是指针，则检查魔数后通过 `f2fs_ifs_private_flags_ptr` 定位到柔性数组中的私有标志位区域进行位操作。Set 操作的分配策略更关键：通过 `f2fs_should_use_buffered_iomap` 判断当前 folio 是否可能进入 iomap 缓冲写路径——若可能（由 `force_alloc` 参数体现），即使该 folio 当前为 0 阶且尚未有任何私有状态，也必须通过 `f2fs_ifs_alloc` 分配统一状态结构，从根本上消除 0 阶 folio 的冲突可能。

## 4.4 压缩文件 Large Folio 读写

压缩文件路径的特殊性在于数据以压缩簇（通常 4 块 16KB）为单位组织，而 Large Folio 的覆盖范围可能与簇边界不重合——一个 64KB 的 folio 可能跨越 4 个 16KB 压缩簇，不同子范围可能分别需要解压填充和直接磁盘读。本节处理压缩文件的两条前台路径：buffered read 和 buffered write。压缩文件的写回路径留至 4.5 节，与普通文件写回对照处理。

### 4.4.1 压缩文件缓冲读

第三章 3.4 节指出，压缩文件读取存在混合 I/O 模型：一个 Large Folio 的不同子范围可能位于压缩簇内（需要"读取压缩块→解压到 folio"的间接路径）或位于非压缩区域（需要"磁盘→folio"的直接读路径）。两条路径的完成时机不同——解压由 CPU 异步执行，bio 完成由中断上下文触发——若无法统一协调，就会出现 folio 在部分子范围仍未就绪时被提前解锁。

本系统为压缩文件缓冲读实现了一套完整的 iomap 适配路径，由三个组件构成：簇感知的 `iomap_begin` 回调、替代 `iomap_readpage_iter` 的内层迭代器、以及统一的完成计数机制。

**iomap_begin：从单簇到多簇。** `f2fs_buffered_read_compress_iomap_begin`（算法 4-5）是压缩读的 iomap 迭代入口。其核心改进在于增加了多簇处理路径。

{{ALG:f2fs_compress_read_iomap_begin}}

在旧的单簇路径中，`iomap->length` 被截断为从 `pos` 到当前簇末尾的长度——iomap 的 `iomap_iter` 外层循环每次迭代最多覆盖一个压缩簇的范围。新的多簇路径在检测到当前 folio 对齐于簇边界且跨越多个簇时（`all_clusters > 1`），遍历全部簇，从 extent cache 或 dnode 获取每个簇的物理块地址信息，填入 `ctx->pblks_arr[]` 数组，最后将 `iomap->length` 设置为覆盖所有连续压缩簇的字节总长度。这意味着外层 `iomap_iter` 一次迭代即可覆盖多个簇的物理范围，减少了迭代次数和映射查询开销。

**内层迭代器：复用外壳，替换内胆。** `f2fs_compress_readpage_iter`（算法 4-6）是本系统对 iomap 通用框架最关键的一处定制。

{{ALG:f2fs_compress_readpage_iter}}

该函数的结构与 iomap 的通用 `iomap_readpage_iter` 同构——两者的外层迭代逻辑和边界处理保持一致，均依赖 `iomap_adjust_read_range` 计算实际读取范围、`iomap_block_needs_zeroing` 处理空洞填零。差异仅在于内层的 I/O 提交策略：通用版本在确定需要读取的区间后直接分配 bio、调用 `bio_add_folio_nofail` 添加数据、立即提交；F2FS 压缩版本则根据 `ctx->compressed_clusters` 的值将子范围分发到两条路径——压缩路径调用 `f2fs_do_read_multi_folios` 将页面暂存到 `compress_ctx` 中延迟提交，等待簇满或预读结束时统一发起磁盘读取并解压；非压缩路径（压缩文件内的非压缩区域）调用 `f2fs_do_read_single_folio_iomap` 直接构建 bio 并提交。

这种"外壳复用、内胆替换"的策略，其必要性来源于压缩数据的延迟提交语义：`iomap_readpage_iter` 的"立即 bio 提交"模式与压缩路径要求的"先聚合后解压"模式在根本上矛盾。直接修改通用 iomap 代码来迁就这一需求将影响所有其他文件系统，因此本系统选择在 F2FS 层实现一个结构同构但内部分发的迭代器。

**完成计数：跨路径统一。** 两条 I/O 路径的完成回调最终均汇集到 `f2fs_iomap_finish_folio_read`。该函数每次被调用时从 `f2fs_folio_state` 的 `read_bytes_pending` 中减去已完成子区间的字节数，仅当计数器归零（或回落至仅剩魔数偏置的 `F2FS_IFS_MAGIC`）时才调用 `folio_end_read` 解锁 folio。无论是解压回调还是 bio 完成回调先到达，在另一个子区间的处理完成之前计数器均不会归零。对于一个跨越 2 个压缩簇和 1 个非压缩区间的 64KB Large Folio，其 `read_bytes_pending` 初始累加了全部子区间的字节总数，三条路径分别完成后各减去对应量，仅当全部完成时 `folio_end_read` 才被调用。

### 4.4.2 压缩文件缓冲写

第三章 3.4 节描述了压缩文件写入中"簇视角"与"Folio 视角"的根本分歧。一种直观方案是以压缩簇为基本写入单位：每次只处理一个簇大小的范围，folio 的阶数受限于簇大小。这种方案下，一个 128KB 的写入请求若横跨 4 个 32KB 簇，将被拆分为 4 次独立的小粒度操作——Large Folio 的聚合优势被完全抵消。

本系统提出一种"从大 Folio 视角出发"的写入算法（算法 4-7），核心思路的反转在于：不再"用多个小 folio 去填充一个压缩簇"，而是"一次性分配一个足够大的 folio，然后用一个或多个压缩簇的数据去填充它"。

{{ALG:f2fs_compress_write_iomap_begin}}

该算法的关键机制是 `read_bytes_pending` 的"偏置-轮询"同步模式。在提交任何异步 bio 之前，先对 `read_bytes_pending` 加一个值为 1 的偏置——此偏置阻止任何 bio 完成回调触发 `folio_end_read`（因为完成回调中的计数器归零判断要求到达 `F2FS_IFS_MAGIC`，加偏置后该条件不满足）。随后 iomap_begin 进入显式轮询循环，检查计数器是否降至 `F2FS_IFS_MAGIC + 1`——"魔数加偏置"意味着所有 bio 已完成，但偏置尚未移除。轮询退出后移除偏置，folio 完全就绪，由其持有方（而非 I/O 完成回调）控制锁的释放。

算法的数据填充分三个阶段。第一阶段处理 head 部分簇——写入起始位置不齐簇边界的首簇，通过 per-block uptodate 位图检查该簇内各子页是否已就绪：若全部就绪则跳过读取，若有任何子页未就绪则通过 `f2fs_read_multi_folios` 发起磁盘读取。第二阶段处理中间完整簇——这些簇将被整个写入覆盖，旧数据没有保留价值：直接调用 `iomap_set_range_uptodate` 将对应 folio 子范围标记为 uptodate，完全跳过磁盘读取。第三阶段处理 tail 部分簇，逻辑与 head 簇相同。对于非压缩区域（`data_blkaddr != COMPRESS_ADDR`），算法回退到普通文件路径。

执行阶段，`f2fs_iomap_get_folio` 检查 `iomap->private` 是否已被准备阶段设置——若是，直接返回已填充就绪的大 folio；否则回退到通用 iomap 的 `iomap_get_folio` 执行标准分配和加锁流程。这意味着 iomap 核心框架在写入数据时拿到的 folio 已经在 `iomap_begin` 中完成了全部压缩数据的异步读取和同步等待。

旧"簇视角"方案将一个 128KB 写入拆为 4 次独立操作，folio 阶数无法超过簇大小（通常 16KB，order-2）。本系统的"大 Folio 视角"算法一次分配 128KB 的 order-5 folio，在单次 `iomap_begin` 调用中完成所有必要簇的数据准备——中间完整簇甚至完全不需要磁盘读取。

## 4.5 脏页写回子范围控制

第三章 3.5 节指出了 Large Folio 条件下脏页写回的两个失控层面。第一，`folio_clear_dirty_for_io` 清除整个 folio 的 `PG_dirty` 标志后，写回路径失去了区分"哪些子块需要实际写回"的信息——一次 4KB 的局部修改可能迫使整个 64KB folio 进入日志追加写，产生 60KB 的无效写放大。第二，写回入口将 folio 拆解为页数组后，逐页调用同一个单页写回函数，该函数以 folio 起始索引作为块坐标查询节点树——对于起始索引为 0 的 64KB folio，同一逻辑块被重复写入 16 次，而子块 1~15 的数据从未到达正确位置。

第三章 3.4 节进一步指出，压缩文件的写回叠加了簇级提交边界问题：一个 folio 跨越多个压缩簇时，每个簇对同一 folio 独立执行加锁/解锁循环，导致锁周期被反复打断——簇 A 的完成回调提前解锁 folio，而 folio 的其他部分仍在等待簇 B 和簇 C 的压缩写出。

### 4.5.1 普通文件写回

本系统用 `f2fs_write_single_data_folio`（算法 4-8）替代原有的逐页写回逻辑，在三个维度上进行控制。

{{ALG:f2fs_write_single_data_folio}}

**子范围提交。** 函数接收字节粒度的 `(start, end)` 范围参数——而非传统写回函数中隐含的"整个 folio"。循环的起止索引 `folio_start_idx` 和 `folio_end_idx` 由调用方传入的范围决定，而非默认覆盖整个 folio。上层的 `f2fs_write_cache_folios` 通过 `f2fs_iomap_find_dirty_range` 从 `f2fs_folio_state` 的 per-block dirty 位图中定位 folio 内的实际脏区，仅将脏子范围传递给本函数。

脏区的判定不再依赖 folio 级的 `PG_dirty` 标志——该标志在 `folio_clear_dirty_for_io` 被整体清除后，子块级别的脏信息保留在 per-block dirty 位图中：由 4.2.3 节的 `iomap_set_range_dirty` 写入，由 `f2fs_iomap_find_dirty_range` 读取。对于一个仅有一个 4KB 子块变脏的 64KB folio，`(start, end)` 仅覆盖该 4KB 范围，循环体只执行一次，写入量为 4KB 而非 64KB——写放大从 16x 降至 1x。

**完成生命周期管理。** 每个子块在提交前通过 `atomic_add(PAGE_SIZE, &fifs->write_bytes_pending)` 递增原子计数器。bio 完成回调 `f2fs_write_end_io` 处理每个 bio 段时执行 `atomic_sub_and_test(fi.length, &fifs->write_bytes_pending)`：仅当计数器归零时才调用 `folio_end_writeback(folio)` 结束该 folio 的 writeback 生命周期。这一机制确保了"一次 bio 完成"不再错误地等同于"一个 folio 完成"——多 bio 场景下，只有全部子块的 bio 均已完成，writeback 状态才被解除。

**重试逻辑下沉。** 当 `f2fs_do_write_data_page` 因 checkpoint 全局锁冲突返回 `-EAGAIN` 时，传统写回路径需要在外层重新获取 folio 并从头重写整个范围。本系统将重试逻辑从外层下沉到子块级别——失败位置之后尚未提交的子块继续提交，已成功提交的子块不受影响。对于同步写回模式（`WB_SYNC_ALL`）和压缩文件写回，重试循环在等待超时后直接跳回当前子块的提交点（`goto retry_multi`），而非回退到 folio 起始索引。这避免了"64KB folio 中前 32KB 已提交成功、后 32KB 因 checkpoint 重试，却需将前 32KB 重新写入"的情况。

表 4-3 对比了传统写回路径与本系统子范围控制路径在 64KB Large Folio 仅 4KB 子块变脏场景下的关键差异。

| 指标 | 传统路径 | 本系统路径 |
|------|---------|---------|
| 实际写入量 | 64KB（整 folio） | 4KB（仅脏子块） |
| 写放大 | 16x | 1x |
| 日志消耗 | 追加 64KB | 追加 4KB |
| 写回完成判定 | 单次 bio 回调 | `write_bytes_pending` 归零 |
| 重试粒度 | 整 folio | 子块级别 |

### 4.5.2 压缩文件写回：簇级扩展

压缩文件的写回复用 `f2fs_write_single_data_folio` 的核心循环，并在两个方向上叠加专属逻辑。

**簇边界限制。** `f2fs_iomap_find_dirty_range` 在检测到压缩文件时，将 `range_end` 截断至当前压缩簇的末尾——`range_end = min(range_end, (cluster_i_end_idx(inode, *range_start >> PAGE_SHIFT) + 1) << PAGE_SHIFT)`。这意味着 `f2fs_write_single_data_folio` 接收到的 `(start, end)` 永远不会跨越压缩簇边界——每次写回调用最多处理一个簇内的子块。跨越多个簇的 Large Folio 被拆分为按簇为单位的多次写回调用，每次调用内部以子范围控制确保只有实际脏块被提交，簇的原子语义得到保持。

**延迟解锁。** 压缩路径设置 `fio.need_lock = LOCK_DONE`，告知底层 `f2fs_do_write_data_page` 不要在 bio 完成后解锁子页所属的 folio。实际的解锁决策由 `f2fs_folio_state` 中第二个额外空间的 `dirty_bytes_pending` 原子计数器驱动：压缩路径在提交整个簇的所有子块后，通过 `atomic_sub_and_test` 检查计数器是否归零，仅在归零时才调用 `folio_unlock`。这一机制消除了跨簇解锁冲突——簇 A 的 bio 完成回调不会提前解锁 folio（因为 `need_lock = LOCK_DONE` 跳过了回调中的解锁步骤），解锁由压缩路径在所有相关子块全部进入完成状态后统一执行。

**空悬 folio 收尾。** `f2fs_write_cache_folios` 在主循环结束后检查最后一个 folio：若其仍处于锁定状态且没有任何子块被提交（`submitted == 0`，例如所有子块均为干净页），则手动调用 `folio_unlock` 和 `folio_end_writeback`，防止其永久空悬。

## 4.6 GC 路径的 Large Folio 适配

第三章 3.2 节指出，GC 搬移路径中存在子块偏移错误：`f2fs_do_write_data_page` 以 folio 的起始页缓存索引作为块坐标来查询节点树——当 GC 请求搬移的逻辑块落在一个 Large Folio 的覆盖范围之内时（例如起始索引为 0 的 64KB folio，GC 要搬移逻辑块号 5），写回函数以索引 0 而非 5 定位节点树条目，将错误块的旧数据写到新段。

本系统在 GC 路径的两个关键点上进行修正，使其与前述所有机制保持一致。

**子范围脏区标记。** GC 后台搬移路径（BG_GC）中的数据页处理函数 `move_data_block` 在标记搬移目标页为脏时，根据 folio 阶数做出分支（算法 4-9）。

{{ALG:move_data_block_gc}}

对 0 阶 folio，沿用原有的全 folio 脏标记（`folio_mark_dirty`）；对高阶 folio，调用 `iomap_set_range_dirty` 以 `bidx`（GC 选中的目标逻辑块号）为偏移，仅标记 folio 内该子块对应的 4KB 范围为脏。这意味着 4.5 节的 `f2fs_write_single_data_folio` 写回路径读取 per-block dirty 位图后仅提交该 4KB 范围——GC 搬移一个 4KB 子块不再触发整个 64KB Large Folio 的写回。

同时，`f2fs_set_folio_private_gcing` 通过 4.3 节 `f2fs_folio_state` 的统一 API 将 `ONGOING_MIGRATION` 标志写入柔性数组的私有标志区，而非直接编码在 `folio->private` 中，消除了与 iomap 指针的语义冲突。

**bio 子页偏移修正。** GC 读路径的 `f2fs_submit_page_read` 在向 bio 添加页面时，以子页在 folio 内的相对索引计算偏移——`(index - folio->index) << PAGE_SHIFT`——确保 bio 层只针对正确的子页范围操作。GC 搬移写路径的 `f2fs_submit_page_bio` 同样使用 `fio->idx` 和 `fio->cnt` 精确指定子范围。这两处修正使 GC 路径的 bio 偏移不再依赖"一个 folio 一个块"的隐含假设。

## 4.7 本章小结

本章以第三章析出的四类问题为线索，给出了一套以 iomap 框架为承载、以 F2FS 专属扩展为补充的 Large Folio 缓冲 I/O 设计方案。各项设计与第三章问题的对应关系如表 4-4 所示。

表 4-4 第三章问题与第四章设计方案对应

| 第三章问题 | 第四章设计 | 核心机制 |
|-----------|-----------|---------|
| 3.2 全路径"一个对象=一个块"假设 | 4.2.1 f2fs_set_iomap 统一转换层 | 块号→字节区间，空洞/NEW_ADDR 精确区分 |
| 3.2 读路径单块映射（16 次调用） | 4.2.2 区间读 iomap_begin + extent cache 合并 | 映射调用 16→1，锁次数同步下降 |
| 3.2 写路径逐块分配 | 4.2.3 区间预分配 + NEW_ADDR 合并 | 元数据操作 N→1，批量空间预留 |
| 3.3 `folio->private` 所有权冲突 | 4.3 f2fs_folio_state 统一状态结构 | 头部兼容+柔性数组扩展+魔数类型识别+force_alloc |

| 3.4 压缩写"簇视角"限制 folio 阶数 | 4.4.2 大 Folio 视角算法 + 跳过全覆盖簇 | folio 阶数不再受限于簇大小 |
| 3.5 写回脏区粗化 + 索引错位 | 4.5.1 per-block dirty 位图 + (start,end) 范围 + write_bytes_pending | 写放大 16x→1x，重试粒度下沉到子块 |
| 3.4 压缩写回跨簇锁冲突 | 4.5.2 簇边界限制 + DEFERRED_UNLOCK 延迟解锁 | 簇语义保持，锁由计数器统一驱动 |
| 3.2 GC 搬移子块偏移错误 | 4.6 子范围脏标记 + bio 偏移修正 | GC 与前台路径共享同一状态框架 |

这些设计共享一套底层基础设施：`f2fs_set_iomap` 提供块到区间的统一转换，`f2fs_folio_state` 提供跨路径的统一状态管理，per-block 位图提供子块级别的脏区和 uptodate 判定。在此基础上，普通文件路径、压缩文件路径、写回路径和 GC 路径各自叠加专属逻辑，但均不脱离这套基础设施——避免了各路径各自形成独立适配方案而导致的维护分叉和语义不一致。
