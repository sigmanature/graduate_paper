## 8. 背景问题的递进式拆解

用户进一步提出一个更清晰的叙事层次，这个判断是成立的，后续应优先采用：

### 第一层问题

传统 4KB 页缓存粒度难以充分利用现代块层和闪存设备所支持的大块连续 I/O 能力。

这一层的重点是：

- 先不要急于把它上升为“全部都是粒度失配”；
- 它本质上是“小粒度缓存对象限制大块 I/O 发挥”的问题；
- Large Folio 的引入首先是为了解决这一层问题。

### 第二层问题

即使引入 Large Folio，F2FS 中仍然存在更深一层的不一致：一个更大的 folio 会覆盖多个文件系统块、多个子区间、多个 bio 或多个完成回调，导致 I/O 提交生命周期、状态管理和完成语义之间难以自然对齐。

这一层的重点是：

- 这时再使用“粒度失配”一词更准确；
- 它不再只是 4KB 与大块 I/O 之间的性能差异；
- 而是 Large Folio 进入 F2FS 后，映射边界、状态边界和完成边界之间的不一致。
- iomap 的引入主要是为了解决这一层问题。

### 对第一章叙事的直接影响

第一章更适合采用“分层递进”而不是“一开始就给一个大而统的失配定义”。

推荐顺序：

1. 应用负载增长，要求更高效的大文件连续 I/O。
2. 传统 4KB 页缓存粒度限制大块 I/O 能力发挥。
3. Large Folio 由此成为第一步解决思路，用于扩大页缓存对象粒度。
4. 但在 F2FS 中，引入 Large Folio 后又暴露出新的跨层协调问题。
5. 这些问题集中表现为映射、状态管理和 I/O 完成生命周期之间的不一致。
6. iomap 因而成为第二步关键技术，用区间映射和统一 I/O 组织方式解决更深层问题。

### 对术语使用的建议

建议不要在第一章最前面就把所有问题统称为“粒度失配”。

更稳妥的做法是：

- 第一层先用“4KB 页缓存粒度限制大块 I/O 发挥”或“粒度过小带来的效率限制”来描述；
- 第二层再正式引出“跨层粒度失配”或“边界失配”；
- 这样可以清楚区分 Large Folio 和 iomap 分别解决什么问题。
7. 人类观点:我依旧认为第一章摘要里面还是没有将粒度失配定义清楚。或者我认为失配的有两层。一层是4KB页和支持更大块io的bio失配。这个和背景里面4KB页无法利用现代块设备大块io能力吻合。另一层是即使使用folio大页(我们可能还得注意术语),那么存在io提交回调生命周期失配。是否都用"粒度失配"这个语言描述存疑。

## 3. 关于第二章应讲什么的明确结论

### 3.1 page、bio、闪存三者关系要不要讲

必须讲，而且必须讲清楚层次。

第二章不能只讲 page，也不能只讲 folio 和 iomap。因为论文后续的问题本质上不是单一数据结构替换，而是跨层粒度不匹配：

- 页缓存层以 `page/folio` 管理缓存对象。
- 文件系统层以 `block/cluster/segment` 管理映射和元数据。
- 块层以 `bio` 聚合并提交 I/O。
- 存储设备侧最终面对的是闪存逻辑地址空间和顺序写、GC、磨损均衡等约束。

如果不先建立这条自上而下的层级关系，后面“为什么需要 folio”“为什么要用 iomap”“为什么 F2FS 适配困难”都没有稳定支点。

### 3.2 F2FS 相关知识要不要讲

必须讲，但不能讲成独立百科全书。

第二章中的 F2FS 背景要服务于后文的三个问题：

1. 为什么 F2FS 比 ext4 更值得研究 Large Folio 的 buffered I/O 适配。
2. 为什么 F2FS 的异地更新、段布局和 GC 会让大粒度缓存对象的适配更复杂。
3. 为什么压缩文件、GC 搬运、写回路径会和普通 buffered I/O 一起构成后文的实现难点。

因此，第二章要讲：

- F2FS 的设计目标与日志结构思想。
- Main Area、NAT、SIT、SSA、Checkpoint 的最小必要背景。
- out-place update 的特性。
- GC 的基本流程及其与有效块搬运、元数据更新的关系。

但是第二章不应展开：

- 具体实现函数。
- 你的补丁如何改某个字段。
- 各路径里的状态兼容细节。

这些内容留给第三章和第四章。

### 3.3 Linux I/O 系统调用栈到 F2FS 栈要不要讲

也必须讲，但要“讲路径，不讲源码细枝末节”。

读者必须知道一次普通文件 buffered read/write 是如何从用户态走到 F2FS 的，否则：

- page cache 在哪里介入不清楚；
- readahead、writeback 在哪里发生不清楚；
- F2FS 与 iomap 的边界也不清楚。

因此第二章建议讲清楚这样一条抽象路径：

`read/write 系统调用 -> VFS -> page cache/filemap -> 文件系统 address_space 操作 -> F2FS 块映射/读写路径 -> bio -> block device -> FTL/flash`

这一节的写法应以“分层关系图 + 文字说明”为主，必要时只摘 1 个非常短的调用链示意，不贴整段源码。

## 4. 第一章建议框架

### 第1章 绪论

#### 1.1 研究背景与意义

建议行文顺序：

1. 端侧 AI、大模型推理、移动应用资源占用增长，推动移动设备对大文件顺序 I/O 和页缓存效率提出更高要求。
2. Android 设备常用 F2FS 作为闪存文件系统，其文件访问性能直接影响应用启动、模型加载和后台维护开销。
3. Linux 页缓存长期以 4KB page 为基本单位，在面对较大连续访问范围时，会带来更多状态管理、映射查询和 I/O 提交开销。
4. 块层已经能够围绕更大粒度组织 bio 和顺序 I/O，但页缓存与文件系统路径仍受到小粒度组织方式约束。
5. 因此，围绕 F2FS 页缓存路径开展 Large Folio 和 iomap 适配研究具有现实意义和工程价值。

#### 1.2 国内外研究现状

只写三类现状，不要散：

1. 大页/页缓存粒度演进相关研究：4KB 页问题、THP 局限、folio 演进。
2. 文件系统块映射与 iomap 相关研究：buffer_head 局限、区间映射框架发展。
3. F2FS 与闪存优化相关研究：日志结构、GC、压缩文件、移动场景应用。

每类现状都必须有“小结句”，最后收束到：

- 现有研究已经分别讨论页粒度优化、区间映射和闪存文件系统设计；
- 但面向 F2FS buffered I/O 路径的 Large Folio 与 iomap 协同适配仍不充分；
- 尤其在 GC、压缩文件和脏页写回场景下，仍存在明显工程空缺。

#### 1.3 研究问题与挑战

本节先正式定义“粒度失配”：

页缓存以 folio 管理内存范围，F2FS 以 block 或 cluster 维护映射和状态，块层以 bio 聚合和提交 I/O。当三者在映射边界、提交边界和完成边界上不能一致表达同一段文件区间时，即形成粒度失配。

然后拆成几个挑战：

1. 映射失配：块映射结果难以直接表达为大粒度页缓存区间。
2. 提交与完成失配：一个 folio 内可能对应多个 bio 与多个完成回调。
3. 状态失配：F2FS 私有状态、iomap 状态、压缩/GC 状态存在复用冲突。
4. 写回失配：局部修改可能引发整 folio 写回，放大日志写入与 GC 压力。

#### 1.4 本文主要工作

这里不要泛写“进行了研究”。直接写 3 到 4 条工作：

1. 分析 F2FS 页缓存路径中的跨层粒度失配问题。
2. 设计基于 iomap 的普通文件区间映射与 Large Folio 状态管理机制。
3. 完成 F2FS 在写回、GC 和压缩文件路径上的 Large Folio 适配。
4. 通过 QEMU 与树莓派平台测试验证方案效果。

#### 1.5 论文组织结构

按章节一句话概述，不展开技术。

#### 2.1 Linux 文件访问路径的层次关系

这一节先给出一张总图，建议画成自上而下四层：

1. 用户态与系统调用层。

#### 2.2 从 Linux buffered I/O 到 F2FS 的数据路径

说明一次 buffered read/write 的抽象调用过程：

- read/write 系统调用经 VFS 进入通用文件读写路径；
- page cache 决定缓存对象的查找、分配、预读和回写；
- F2FS 根据文件逻辑块建立块映射；
- 块层将请求组织为 bio 并提交到底层设备。

注意：

- 这里讲“路径职责”，不贴大段源码；
- 只需要让读者知道 page cache 与 F2FS 的边界在哪里。

#### 2.3 F2FS 的基本结构与闪存友好设计

这一节讲最必要的 F2FS 背景：

- F2FS 的日志结构思想；
- out-place update；
- Main Area、NAT、SIT、SSA、Checkpoint 的基本作用；
- 为什么这些结构会让映射、写回和 GC 不只是“块读写”问题。

#### 2.4 F2FS 的垃圾回收与压缩文件机制

这一节只讲和后文直接相关的两件事：

这节写完后，读者应当明白：F2FS 的复杂性不是只有“块映射”，还包括 GC 和压缩路径。
