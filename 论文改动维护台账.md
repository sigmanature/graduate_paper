### 1.2 第四章 4.1：开头改为架构原则，不再说“以第三章问题链为输入”

**删除：**

**建议替换：**

> F2FS Large Folio buffered I/O 的实现采用“通用框架复用、F2FS 回调接入、共享状态支撑、专属路径扩展”的结构。图 4-1 给出了各组件之间的关系：上层读写与写回流程尽量复用 iomap 已有迭代器和 buffered I/O 入口；F2FS 通过 `iomap_begin` 回调和写回入口接入自身块映射、空间分配和子范围提交逻辑；公共状态由 `f2fs_set_iomap`、`f2fs_folio_state` 和 per-block 位图承担；GC 与压缩文件路径在此基础上叠加各自的边界修正和生命周期控制。

**后续展开建议：**

> 在前台 buffered read/write 路径中，F2FS 不改动 `iomap_readahead` 和 `iomap_file_buffered_write`，而是在 `f2fs_buffered_read_iomap_begin` 与 `f2fs_buffered_write_iomap_begin` 中提供区间映射和预分配结果。写回路径由 `writeback_iter` 遍历脏 folio 后进入 `f2fs_write_single_data_folio`，以 folio 内部的实际脏子范围为单位提交写回。上述路径共同依赖 `f2fs_set_iomap` 完成块映射到 iomap 区间的转换，并依赖 `f2fs_folio_state` 保存 per-block 位图、F2FS 私有标志和完成计数。

> GC 与压缩文件路径不另建独立框架，而是在公共机制上增加路径约束：GC 搬移保留单块语义，但通过子范围 dirty 标记和 bio 偏移修正定位 Large Folio 内部的目标块；压缩读复用 `iomap_iter` 的外层推进逻辑，但以 `f2fs_compress_readpage_iter` 替换通用读迭代器内部的提交策略；压缩写复用 buffered write 的区间准备流程，并在完整覆盖的中间簇上跳过旧数据读取。所有路径最终仍经 bio 层提交到底层块设备。

## 4. I/O 生命周期 → pending 计数 → 4.3 重构线

### 4.1 核心叙事判断

第二章已经讲了 bio 完成回调的异步性：提交 bio 的路径已经返回，完成回调稍后在中断或软中断上下文中更新页缓存对象状态。这个事实应当直接导向第四章：Large Folio 的一个 folio 可能被拆成多个子范围、多个 bio、多个压缩簇，因此不能再由一次回调直接解除整个 folio 的读锁或写回状态。

### 4.2 4.3 开头建议写法

> 在 Large Folio 条件下，页缓存对象的状态需要拆成两个层次理解：folio 级标志仍然描述整个对象是否 locked、dirty 或 writeback；但一个 folio 内部包含多个文件系统块，不同子块可能处于不同的 uptodate、dirty 或 I/O 未完成状态。iomap 为此引入 `iomap_folio_state`，即 folio 内部子区间状态对象。该对象挂接在 `folio->private` 上，内部用逐块位图记录 folio 内哪些文件系统块已经 uptodate 或 dirty，并用 pending 字节计数汇总多个异步 I/O 完成事件。只有当相关 pending 计数归零时，读路径才能解锁 folio，写回路径才能结束 writeback。

接着再进入 F2FS 冲突：

> 这个状态对象解决了 Large Folio 内部子区间状态表达问题，却与 F2FS 原有 `folio->private` 用法发生冲突。F2FS 原本也使用该字段保存 GC 迁移、原子写等私有标志；如果同一字段一会儿被解释为指针，一会儿被解释为整数标志，跨路径流动的 folio 就会出现语义冲突。本文因此设计 `f2fs_folio_state`：它保留 `iomap_folio_state` 的头部布局，使 iomap 能继续按原方式访问位图和 pending 计数，同时在后续空间中扩展 F2FS 私有标志和压缩路径所需的脏 pending 计数。

### 4.3 术语替换规则

| 原模糊说法 | 建议替换 |
|---|---|
| 统一状态管理 | folio 内部子区间状态表达 |
| 统一状态结构 | F2FS 兼容扩展型 folio 状态对象 |
| 完成计数 | 未完成读/写回字节计数 |
| 压缩 pending | 压缩上下文待消费脏字节计数 |
| 私有字段冲突 | `folio->private` 字段所有权冲突 |
