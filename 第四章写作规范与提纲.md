# 第四章 写作规范与提纲

## 一、架构图设计原则（核心约束）

**禁止画成三层蛋糕图。** `f2fs_set_iomap`（函数）、`f2fs_folio_state`（结构体）、`f2fs_write_single_data_folio`（函数）三者不是上下层叠关系，而是**并列机制组件**。

正确架构图是**组件关系图**：
- 左侧：iomap 三个入口钩子（readahead / buffered_write / writepages）
- 中间：F2FS 回调实现（iomap_begin / iomap_end）+ 三个核心机制组件
- 右侧：GC 和压缩两条扩展路径（复用中间组件 + 各自专属逻辑）
- 底部：bio 提交层

## 三、每个实现模块的固定写作定式

### 定式 A：数据结构/基础机制

```
1. 问题来源（一句话，不超过两行）
   引用第三章的具体节号，说明是什么具体问题驱动了这个设计。

2. 设计目标
   这个机制要达成的核心效果，一句话。

3. 核心结构与算法
   - 给出关键数据结构定义（代码块）
   - 给出关键函数/宏的伪代码或核心逻辑描述（代码块）
   - 说明关键字段的含义和用途

4. 优化点/设计要点分析
   - 用数字说话：映射次数从 N 降到 1，锁次数从 N 降到 1
   - 说明哪种不变式得到了保证
   - 说明旧路径到新路径的迁移路径

5. 问题回应（一句话）
   回应当初第三章提出的哪个问题，解决到什么程度。
```

## 四、小节命名规则

1. **功能优先，技术实现细节后置**：标题用"缓冲读路径"而非"f2fs_buffered_read_iomap_begin"
2. **共用组件前置**：`f2fs_set_iomap` 作为 4.2.1 独立小节，后续路径直接引用
3. **压缩文件独立成节**：4.5 专门处理压缩文件，因为簇级语义足够特殊
4. **脏页写回包含压缩**：4.4.2 在 4.4.1 普通写回基础上叠加压缩写回

## 五、句式禁令（第四章专用）

在第四章写作中，禁止使用以下句式：

| 禁止句式 | 原因 | 替代 |
|---------|------|------|
| "本节的设计回应了第三章 X 节提出的……" | 为回扣而回扣。前面已经把问题讲清楚了，不需要重复宣告 | 在"问题来源"里写一次就够了。实现部分只说事实，不说"回应" |
| "这一机制不仅解决了 X，还同时解决了 Y 和 Z" | 堆砌"不仅……而且" | 分别说明解决 X 的效果和解决 Y 的效果 |
| "从上述分析可以看出" | AI 万能过渡句 | 如果前面分析清楚，结论直接写 |
| "需要指出的是"、"值得注意的是" | 无信息量 | 删除。需要指出的就直接指 |
| "换言之"、"简言之" | AI 高频填充词 | 如果前面不清楚，重写；如果清楚，删掉 |
| "具有较强的" | 模糊断言 | 说具体原因或删除 |
| "在……方面具有……优势" | 广告式语言 | 说具体数字或具体效果 |
