# 第四章 写作规范与提纲（v2）

## 一、架构图 v2（最终版）

详见 `架构图_v2.md`。与 v1 的关键差异：

| 改动 | v1 | v2 |
|------|----|----|
| iomap 框架 | 散落在入口描述中 | **独立顶层框**，仅列被复用的组件：`iomap_readahead` / `iomap_file_buffered_write` / `writeback_iter` |
| 写回入口 | `iomap_writepages` | `writeback_iter`（实际代码使用的迭代器） |
| 压缩文件 | 单一方块 | **拆为两块**：压缩读矩形 + 压缩写矩形 |
| 压缩读 | 无 | 三层标注：复用 `iomap_iter` 数据结构 → 复刻 `f2fs_compress_readpage_iter` 替代 `iomap_readpage_iter` → 扩展多簇物理块信息一次性获取 |
| 压缩写 | 无 | 两层标注：复用 `f2fs_buffered_write_iomap_begin` → 扩展跳过完全覆盖的中间簇 |
| 核心机制③ | `read/write_bytes_pending` | 加入 `dirty_bytes_pending`（压缩写回延迟解锁用） |
| 结构体名 | `f2fs_iomap_folio_state` | `f2fs_folio_state`（与实际代码一致） |

## 二、第四章重写提纲（v2）

```
第四章 F2FS Large Folio 缓冲 I/O 设计与实现

4.1 总体架构
    [图 4-1：组件关系图 v2]
    架构总览：iomap 框架复用层 → F2FS 三个 iomap_begin 回调 → ①②③核心机制 → GC/压缩扩展 → bio 层

4.2 普通文件区间映射 I/O
    4.2.1 块映射到 iomap 区间的转换：f2fs_set_iomap
        算法伪代码：f2fs_set_iomap.alg
        关键点：F2FS_MAP_MAPPED/NEW_ADDR/NULL_ADDR 三类状态的区间翻译
               NODNODE 空洞 vs 普通空洞的区分
               IOMAP_F_MERGED/IOMAP_F_NEW/IOMAP_F_DIRTY 标志传递

    4.2.2 缓冲读路径
        算法伪代码：f2fs_read_iomap_begin.alg
        关键点：字节区间→块区间转换 → f2fs_map_blocks(F2FS_GET_BLOCK_IOMAP)
               extent cache 快速路径 + dnode 遍历 + map_is_mergeable 块合并
               对比：原生 16 次逐块映射 → 本文 1 次区间映射

    4.2.3 缓冲写路径
        算法伪代码：f2fs_write_iomap_begin.alg
        关键点：f2fs_map_blocks_iomap 已有映射 + f2fs_map_blocks_preallocate 空洞预分配
               NEW_ADDR 合并条件：m_pblk==NEW_ADDR && blkaddr==NEW_ADDR
               iomap 框架收益：自动 folio 生命周期 + iomap_set_range_dirty 精准脏区标记

4.3 F2FS 扩展型 folio 内部状态对象
    算法伪代码：f2fs_ifs_alloc.alg
    关键点：头部兼容 iomap_folio_state（前三成员完全一致）
           柔性数组扩展：[iomap位图|F2FS私有标志|dirty_bytes_pending]
           魔数 F2FS_IFS_MAGIC 类型识别
           三路分配：为空→新建 / 已是己方→直接返回 / 是iomap_folio_state→拷贝迁移
           force_alloc 机制处理 0 阶 folio 的 GC/写冲突

4.4 压缩文件 Large Folio 读写
    4.4.1 压缩文件缓冲读
        算法伪代码：f2fs_compress_read_iomap_begin.alg + f2fs_compress_readpage_iter.alg
        关键点：
        - iomap_begin 双路径：单簇（旧）vs 多簇（新）。多簇路径遍历 all_clusters，
          从 extent cache 或 dnode 获取每簇物理块地址，填入 pblks_arr[]，
          一次返回覆盖所有簇的 iomap->length
        - f2fs_compress_readpage_iter 的设计决策：
          外壳复用 iomap（iomap_adjust_read_range + iomap_block_needs_zeroing）
          内胆替换（压缩→延迟聚合 f2fs_do_read_multi_folios；非压缩→直接提交）
          为什么不能直接用 iomap_readpage_iter：通用版本立即构建 bio 提交，
          压缩需要先暂存到 compress_ctx 等簇完整后统一读取并解压
        - read_bytes_pending 统一完成计数：解压回调和 bio 回调汇集到
          f2fs_iomap_finish_folio_read，计数器归零才解锁

    4.4.2 压缩文件缓冲写
        算法伪代码：f2fs_compress_write_iomap_begin.alg
        关键点：
        - 大 Folio 视角算法：先分配覆盖全部簇的大 folio，以多个簇数据填充
        - read_bytes_pending 偏置-轮询同步机制：+1 偏置阻止提前唤醒，
          轮询等待 MAGIC+1，移除偏置后 folio 完全就绪
        - 三阶段处理：head 部分簇（read-modify-write，uptodate 全则跳过读）
          → 中间完整簇（直接 iomap_set_range_uptodate，不读磁盘）
          → tail 部分簇（同 head）
        - 对比初赛"簇视角"方案：128KB 写入→4 次独立小操作→folo 阶数≤簇大小
          决赛"大 Folio 视角"：128KB 写入→1 次 order-5 分配→folio 阶数不限

4.5 动态大页脏页写回
    4.5.1 普通文件动态大页脏区间写回
    4.5.2 压缩文件动态大页写回与延迟解锁
        关键点：
        - 簇边界限制：f2fs_iomap_find_dirty_range 对压缩文件截断 range_end 到簇尾
        - DEFERRED_UNLOCK 延迟解锁：压缩路径 need_lock=LOCK_DONE 跳过回调解锁，
          由 dirty_bytes_pending 原子计数器在簇全部完成后统一驱动 folio_unlock
        - 收尾：f2fs_write_cache_folios 主循环后处理空悬 folio（未提交且仍锁定的）

4.6 GC 路径的 Large Folio 适配
    算法伪代码：move_data_block_gc.alg
    关键点：
    - BG_GC 路径：folio_order==0 → folio_mark_dirty（旧）；>0 → iomap_set_range_dirty(folio, bidx << PAGE_SHIFT, PAGE_SIZE)
    - GC 读偏移修正：f2fs_submit_page_read 中 (index - folio->index) << PAGE_SHIFT
    - GC 写偏移修正：f2fs_submit_page_bio 中 fio->idx 和 fio->cnt
    - GC 读/写偏移修正：f2fs_submit_page_read 和 f2fs_submit_page_bio 使用子页偏移
      修正为 bio 仅绑定正确子页范围

4.7 本章小结
    全章对应关系汇总表：
    3.2 全路径单块假设 → 4.2 f2fs_set_iomap + 读/写 iomap_begin
    3.3 私有字段冲突   → 4.3 f2fs_folio_state + f2fs_ifs_alloc
    3.4 压缩簇边界     → 4.4 压缩读（多簇+复刻迭代器）+ 压缩写（大Folio视角）
    3.5 写回粒度失控   → 4.5 子范围写回 + write_bytes_pending
    3.2 GC 搬移偏移   → 4.6 子范围脏标记 + bio 偏移修正
```

## 三、伪代码规约（新增）

### 格式标准

使用 word-thesis-pseudocode 的 `.alg` DSL 格式：

```text
@algorithm id=xxx chapter=4 index=N title="中文标题"

\Input 参数列表
\Output 返回值
主体逻辑（4 空格缩进）
\If 条件
    \State 操作
\EndIf
\For 循环
    \State 操作
\EndFor
\Return 结果
```

### 取舍原则

| 保留 | 删除 |
|------|------|
| 函数输入输出、核心分支条件 | `WARN_ON_ONCE`、`f2fs_bug_on` 等调试断言 |
| 区间的计算逻辑（offset/len 转换） | `spin_lock_irqsave` 并发原语 |
| 类型/标志的决策条件 | `kzalloc`/`kfree` 内存分配细节 |
| 计数器的增减语义 | `printk`/`#ifdef DEBUG` 调试代码 |
| 关键的不变式保证 | 错误处理中不影响主逻辑的释放路径 |

| 编号 | id | 标题 | 所属节 |

## 四、写作规则修订

### 规则 1：禁止显式"定式前缀"

`**问题来源。**`、`**核心设计。**`、`**问题回应。**` 这些前缀是**内部写作指导标签**，不应出现在论文正文中。论文正文应自然叙述，段落之间靠逻辑递进衔接，不靠模板填空。

### 规则 2：禁止末尾"回扣段"

每节末尾不再设独立的"问题回应"段落。第三章到第四章的对应关系通过：
- 节首一句话交代问题上下文（自然提及，不显式引用节号）
- 正文中的数字对比（如"映射调用从 16 降至 1"）本身就是对问题的回答
- 4.7 章小结一次性汇总全章对应关系

### 规则 3：伪代码替代 C 源码

所有代码块替换为算法伪代码（`.alg` DSL），正文中通过 `{{ALG:xxx}}` 占位符插入。

### 规则 4：句式禁令（延续 v1）

| 禁止句式 | 替代 |
|---------|------|
| "本节的设计回应了第三章 X 节……" | 不需要。问题来源写一次即可 |
| "这一机制不仅解决了 X，还同时解决了 Y 和 Z" | 分别说明效果 |
| "不是……而是……" | 直接陈述是什么，不否定 |
| "不只是……还……" | 去掉"不只"，直接用"涉及/包括" |
| "从上述分析可以看出" | 直接写结论 |
| "需要指出的是"、"值得注意的是" | 删除 |
| "换言之"、"简言之" | 删除或重写 |
| "具有较强的" | 说具体原因 |
| "在……方面具有……优势" | 说具体数字 |
| 显式章节号回扣（"回应了第三章 X 节"） | 不执着显式回扣句，通过不断审读达到逻辑通顺、上下映衬 |

### 规则 5：小节命名（延续）

- 标题用功能名而非函数名（"缓冲读路径"而非"f2fs_buffered_read_iomap_begin"）
- 共用组件前置（f2fs_set_iomap 在 4.2.1，后续直接引用）
- 压缩文件独立成 4.4，压缩读/写各一个子节
- 脏页写回独立成 4.5，内部普通/压缩对照
