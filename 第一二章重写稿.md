# 第一章 绪论

## 1.1 研究背景与意义

近年来，移动终端上的应用形态正在发生明显变化。一方面，社交、办公、影像处理等应用持续增加本地数据规模，应用启动、资源装载和后台更新过程对文件系统吞吐能力提出了更高要求；另一方面，端侧人工智能模型逐步进入实际应用场景，大语言模型、多模态模型和视觉推理模型通常以较大文件形式存放在本地，其加载、缓存和增量更新过程都会产生持续的大文件顺序 I/O 需求。与传统以小文件和零散访问为主的移动应用相比，这类负载更强调连续数据读取能力、页缓存（page cache，内核用于缓存文件数据的内存缓冲区）命中效率以及后台写回开销控制。文件系统如果仍长期围绕较小粒度组织缓存和 I/O，往往难以充分发挥当前闪存设备的顺序访问能力。，往往难以充分发挥当前闪存设备的顺序访问能力。

在 Android 设备中，F2FS（Flash-Friendly File System，闪存友好文件系统）已成为常见的本地文件系统之一。F2FS 面向 NAND 闪存和基于闪存转换层（Flash Translation Layer，FTL）的块设备设计，强调顺序追加写入、冷热数据分离以及垃圾回收效率，对移动终端场景具有较强适配性。随着移动设备存储带宽和容量持续提升，F2FS 不仅承担普通应用文件访问，还直接影响模型文件装载、媒体资源处理和后台维护任务的整体开销。因此，围绕 F2FS 文件访问路径的缓存粒度和 I/O 组织方式展开研究，具有较强的现实背景和工程意义。

在 Linux 内核中，页缓存通过一种称为页描述符（`struct page`）的数据结构管理每一个被缓存的数据单元。每一个页描述符对应 4KB 的物理内存，记录该内存页的引用计数、映射归属、回写状态等元数据。页缓存每缓存 4KB 文件数据，就需要分配和维护一个独立的页描述符，这意味着文件访问的缓存管理粒度天然锁定在 4KB。当应用需要读取一段较大的连续文件区间时——例如 64KB 的模型参数块——页缓存必须围绕十六个 4KB 页描述符逐一建立状态、逐一查找映射、逐一组织 I/O 提交。这个过程中，元数据维护、锁操作和映射查询的开销与需要处理的页描述符数量成正比，而不是与数据的实际意义成正比。

而在页缓存之下，块层（block layer，负责将读写请求组织为块 I/O 请求 bio 并提交到存储设备的内核子系统）早已具备承载更大粒度 I/O 的能力。一个 bio 可以将多个物理连续的块合并为一次提交，闪存设备本身也倾向于接收较大的顺序访问请求。然而，只要页缓存仍以 4KB 页描述符为对象组织缓存和状态，上层路径的调度和控制就始终受限于小粒度对象，块层和设备的连续 I/O 能力无法在上层得到充分利用。这就是本文面对的第一层问题：4KB 页缓存粒度限制了对底层大块连续 I/O 能力的利用。

围绕上述问题，一个自然的思路是使用更大的内存页。Linux 内核提供透明大页（Transparent Huge Pages，THP）机制，在满足条件时自动将多个连续的 4KB 页合并为 2MB 的大页，以减少页描述符数量和地址转换开销。但 THP 主要面向匿名内存（anonymous memory，即进程的堆、栈等不与文件关联的私有内存），其对文件页缓存路径的适用性很有限。首先，2MB 的固定粒度对于文件访问而言过大——一次通常的文件读写在数十到数百 KB 量级，强制分配 2MB 会造成严重的内部碎片：应用实际只使用其中少量 4KB 子区域，却需要为整个 2MB 分配物理内存和维护页表映射。其次，维持 2MB 大页需要系统频繁执行内存压缩（compaction）来寻找物理连续的 2MB 空间，在长期运行的移动设备上，物理内存碎片化使得这类高阶分配的成功率持续下降，系统在大页折叠与回退到 4KB 小页之间反复切换，反而引入额外的 CPU 开销和延迟抖动。正因为透明大页的粒度固定、面向场景单一且缺乏对文件访问模式的自适应能力，它并不能作为大型文件页缓存路径的直接解决方案。

为此，Linux 内核近年来逐步引入 folio 机制。folio 在类型层面对页描述符做一个关键改进：一个 folio 必定对应一个完整的、可独立管理的缓存对象，而不会指向一个已聚合页组中的附属页。这一类型保证消除了内核代码中对页指针归属的反复判断，使页缓存可以自然地管理不同大小的缓存对象。当文件系统声明自身支持更大粒度的缓存对象后，页缓存可以在合适的访问模式下分配阶数（order）大于等于 1 的 folio——即可变阶数动态大页（Large Folio）。阶数决定了 folio 的大小：order=2 为 16KB（4 个连续 4KB 页），order=4 为 64KB（16 页），依此类推。与 THP 不同，Large Folio 的阶数不是固定预设的，而是由页缓存在运行时根据文件访问的连续范围和内存空闲状况动态决定。因此，Large Folio 首先回应的是第一层问题：通过扩大页缓存对象粒度，从根本上减少缓存对象数量、锁操作次数和映射查询频率，使页缓存先具备承接大块连续文件访问的能力。

然而，Large Folio 只是让页缓存对象变大，F2FS 的内部路径并不会因此自动适配。F2FS 的普通文件读写、脏页写回、GC 搬移以及压缩文件处理，全部路径在实现上隐含了一个共同前提：一个页缓存对象对应一个文件系统块（4KB）。文件系统以块为单位组织元数据和映射，块映射接口以块为单位返回结果，GC 搬移以块为单位加锁标记，压缩机制以簇（通常 2 至 4 个块）为单位收集页面——所有逻辑都不曾预见到一个缓存对象内部同时包含多个块的情况。

当页缓存将一个 64KB（16 块）的 Large Folio 递交给 F2FS 时，上述前提在三条线索上同时破裂。第一，映射边界不一致：F2FS 的块映射机制以逻辑块号为单位返回单个块的物理地址和连续长度，而 Large Folio 覆盖的是字节区间——一个 folio 需要多次块映射调用才能填满，且每次返回的连续长度很可能小于 folio 的覆盖范围。第二，状态语义冲突：F2FS 在页缓存对象的私有存储空间中直接编码 GC 迁移状态和原子写标记等私有信息；而 iomap 框架期望同一存储空间保存的是 per-folio 的逐块状态位图指针。两条路径对这个字段赋予互不相容的解释——一方写整数标志，另一方按指针解引用——一旦同一对象先后经过两条路径，就会导致所有权冲突，轻则状态误判，重则内核崩溃。第三，I/O 完成生命周期不一致：一个 Large Folio 可能对应多个 bio，各 bio 的完成时间各不相同，传统路径中"一次 bio 完成即一个页完成"的直觉不再成立——读完成判断和写回收尾时机需要跨多个异步事件协调。

上述三类问题是 F2FS 在接纳 Large Folio 时必然面临的技术障碍，而 iomap 框架为此提供了关键的应对基础。iomap 的核心设计思想是以字节区间为单位描述文件逻辑到物理存储的映射，并以此区间作为后续缓冲读写、写回和直接 I/O 的统一 I/O 提交粒度。这一设计同时解决了两个矛盾：第一，它使文件系统能够一次性返回与 Large Folio 覆盖范围相匹配的连续映射区间，无需逐块反复查询；第二，它使 bio 的构建从"每个块一次提交"转变为"每个区间一次提交"，让块层的大块连续 I/O 能力得以在文件系统层被有效利用。此外，iomap 内置的 per-folio 逐块状态追踪和原子完成计数机制，已经回应了 I/O 完成生命周期不一致问题——通过在每个 folio 上维护未完成字节数，确保多个 bio 全部完成后才判定读完成或写回结束。

然而，仅仅"使用"iomap 并不足以适配 F2FS 的全部路径。iomap 是通用框架，而 F2FS 带有自身的特殊性——GC 搬移中的私有状态标记、压缩文件的簇级处理语义、原子写标志等——这些机制原本直接编码在页缓存对象的私有存储空间中，与 iomap 期望利用同一空间保存 per-folio 位图状态指针的设计相冲突。因此，在 iomap 提供区间映射和完成计数的通用基础之上，本文还需要对其进行 F2FS 专属扩展：设计一个统一状态结构，将 F2FS 私有标志、iomap 位图状态和压缩路径的 pending 计数纳入同一解释框架，并将这一结构同步覆盖到 GC 搬移和压缩文件的全部路径中。这构成本文的第三层工作——在 Large Folio 扩大缓存对象、iomap 提供区间化 I/O 组织之后，通过扩展 iomap 实现 F2FS 的完整兼容。

基于这一三层递进关系，本文的技术路线可以概括为：第一层，通过 Large Folio（可变阶数动态大页）扩大页缓存对象粒度，缓解 4KB 过小对连续 I/O 能力的限制；第二层，通过 iomap 将 I/O 组织方式从逐块映射提升为字节区间映射，使大粒度页缓存对象能够匹配大粒度 I/O 提交；第三层，通过扩展 iomap 的统一状态结构和 F2FS 专属适配，解决私有字段兼容、GC 路径协同和压缩文件支持等 F2FS 特有的工程难题。



## 1.2 国内外研究现状

### 1.2.1 页粒度与大页机制相关研究

围绕内存页粒度的研究，长期关注的核心问题在于小页的灵活性与大页的映射效率之间如何平衡。早期研究指出，4KB 页面虽然有利于控制外部碎片，但在较大工作集和多级页表环境下，会缩小 TLB 覆盖范围并增加地址转换开销。随着内存容量增长和访问范围扩大，围绕大页、superpage 以及地址转换成本的研究不断增多，相关工作普遍认为，在连续访问占比较高的场景中，扩大基本管理粒度有助于降低页表项数量和相关元数据维护开销。

在 Linux 系统中，透明大页曾被视为缓解小页管理成本的重要手段。然而，透明大页主要围绕匿名内存展开，固定 2MB 粒度在长期运行系统中容易受到内存碎片、高阶页分配困难以及内部碎片等问题影响。对于文件页缓存路径而言，透明大页的粒度过大且控制方式有限，难以稳定适应复杂文件访问模式。近年来，内核社区逐步推动 folio 机制取代传统以 `struct page` 为核心的部分接口，试图以更明确的类型表达单页或复合页整体，并为页缓存引入更灵活的大粒度缓存对象。这为面向文件系统的 Large Folio 支持奠定了基础。

总体来看，已有研究已经从体系结构和内核实现两个层面说明了传统 4KB 页面在大范围连续访问场景中的局限。但现有大页研究仍主要关注内存管理侧的地址转换和碎片控制，未深入探讨页缓存粒度变化对文件系统 I/O 路径组织方式的影响。同时，块层大粒度 bio 和顺序 I/O 的研究聚焦于设备侧请求合并与调度策略，未向上延伸到页缓存对象粒度与文件系统映射语义的协同层面。简言之，页缓存层的粒度演进与文件系统层的 I/O 组织之间，仍缺少系统化的衔接研究。

### 1.2.2 文件系统映射与 iomap 相关研究

在传统 Linux 文件系统实现中，buffer_head 长期用于描述页内文件系统块与底层设备块之间的关系。该模型在 page 与 block 粒度一致的条件下具有较强通用性，但随着页缓存对象和 I/O 请求粒度增大，其局限逐渐暴露。一方面，buffer_head 强依赖“页内多个小块状态对象”的组织方式；另一方面，路径上的读写、写回和完成处理也往往建立在逐块遍历基础上。当文件系统需要返回更大范围的连续映射时，这种面向单块对象的模型会增加状态维护和路径组织成本。

为解决这一问题，iomap 框架提出了区间映射思想。与传统按块返回结果的接口不同，iomap 直接以文件逻辑字节区间为单位描述底层映射关系，并通过统一的迭代器与回调接口支持 buffered I/O、direct I/O 和 writeback 等路径。XFS 是较早全面采用 iomap 的文件系统，ext4、ext2 等文件系统也逐步围绕 direct I/O 或 buffered I/O 迁移到 iomap 路径。已有实践表明，区间化映射方式能够减少对每块状态对象的依赖，更适合与 Large Folio、大块顺序 I/O 和大块设备访问能力相结合。

现有研究与文档已经说明了 iomap 的抽象优势和在主流文件系统中的推广趋势，但面向 F2FS 的协同适配仍处于持续演进阶段。具体而言，如何在 F2FS 的块级映射与 iomap 字节区间语义之间建立稳定的转换路径，如何在 iomap per-folio 状态框架中容纳 F2FS 的 GC 和原子写等私有语义，以及如何使写回和压缩路径的完成判断适配 iomap 的原子计数机制，这三个递进层面的工程问题仍缺乏系统化的解决分析。

### 1.2.3 F2FS 与闪存友好文件系统相关研究

F2FS 最初作为面向 NAND 闪存设计的日志结构文件系统提出，其设计目标是在块设备抽象之上尽量保持顺序追加写入，降低随机覆盖更新带来的写放大，并借助冷热数据分离、多日志和垃圾回收机制改善闪存存储效率。相关研究和官方文档对 F2FS 的 NAT、SIT、SSA、Checkpoint 以及段、节、区等空间组织方式进行了较系统说明，也讨论了其在移动设备和通用闪存场景中的应用价值。

除基础布局外，已有研究还关注了 F2FS 垃圾回收、压缩文件、热冷数据分离和块分配策略等问题。这些机制提升了 F2FS 对闪存设备的适应能力，但也使其文件访问路径较普通文件系统更复杂。例如，异地更新意味着写回并非原地覆盖，垃圾回收需要读取并搬移有效块，压缩文件则引入了簇级处理语义。这些特性决定了 F2FS 在面对更大粒度页缓存对象时，不能简单沿用传统单页或单块路径。

总体而言，已有工作已经分别说明了大页机制、区间映射框架和 F2FS 闪存优化设计的必要性，但对本文所关注的三层递进问题尚未形成完整回答：现有研究证明了 4KB 粒度在连续访问场景中的局限，也为 Large Folio 提供了基础；iomap 框架展示了区间化 I/O 组织的通用优势；但针对 F2FS 场景下如何将 Large Folio、iomap 与 F2FS 自身的 GC、压缩和私有状态机制协同适配，尚缺乏系统化的实现分析。

## 1.3 研究问题与挑战

如 1.1 节所述，本文面对三层递进的技术问题。

第一层问题是页缓存对象粒度过小。传统 4KB 页描述符在连续文件访问场景下将较大区间拆解为大量小粒度对象，导致页缓存元数据维护、映射查询和锁操作开销与对象数量成正比。Large Folio 通过可变阶数扩大页缓存对象粒度，直接回应这一层问题。

第二层问题是 Large Folio 进入 F2FS 后，文件系统原有的逐块 I/O 组织方式与大粒度页缓存对象之间不再匹配。F2FS 的块映射、脏页标记和写回提交均以单个块为单位，而 Large Folio 覆盖的是一个字节区间——两者所需要的 I/O 提交粒度完全不同。iomap 框架以字节区间为基本 I/O 提交粒度，并内部维护 per-folio 的逐块状态追踪和原子完成计数，为回应这一层问题提供了通用基础。

第三层问题是 iomap 作为通用框架，并不天然兼容 F2FS 的私有机制。F2FS 在页缓存对象的私有存储空间中直接编码 GC 迁移标记、原子写状态和压缩上下文等私有信息，而 iomap 期望同一空间保存 per-folio 位图状态指针——两条路径对同一字段的解释冲突直接导致所有权竞争和崩溃风险。此外，F2FS 的压缩文件按簇组织 I/O，GC 在后台搬移中需要跨路径继承页缓存状态，这些 F2FS 专属路径需要 iomap 进行针对性的扩展适配。

基于上述三层递进关系，本文面临以下具体挑战。

第一，F2FS 的块级映射结果需要被稳定转换为 iomap 所需的字节区间语义。这不仅涉及块号到字节地址的单位换算，更需要处理空洞、新分配区间和压缩簇边界等不同映射类型在区间层面的正确表达。若缺乏这一转换层，Large Folio 无法在普通读写路径中获得与其覆盖范围相匹配的映射结果。

第二，F2FS 的私有语义与 iomap 的 per-folio 状态需要在同一对象上共存。这要求设计一个统一状态结构，同时容纳 F2FS 的 GC 标志和原子写标记、iomap 的逐位 uptodate/dirty 位图以及压缩路径的 pending 计数，并将该结构同步覆盖到 GC 搬移和压缩文件的全部路径中，避免因状态存放位置或解释方式不一致导致的行为回归。

第三，Large Folio 条件下的写回和完成生命周期需要跨多事件协调。一个 folio 可能对应多个 bio，各 bio 完成时间不同，传统路径中一次 bio 完成即判定读完成或写回结束的假设不再成立。在 F2FS 的日志结构下，若写回边界控制不当，局部修改可能被放大为整个 folio 的追加写入，产生额外的日志空间消耗和 GC 压力。

## 1.4 本文主要工作

围绕上述研究问题，本文的主要工作如下。

第一，分析 F2FS 页缓存路径中因 Large Folio 引入而暴露的三层递进问题：4KB 粒度过小对连续 I/O 能力的限制、逐块 I/O 组织与字节区间页缓存对象之间的不匹配、以及 F2FS 私有语义与 iomap 通用框架之间的兼容冲突。本文从这些层面出发，归纳了三类问题在普通文件读写、压缩文件处理、脏页写回和 GC 搬运场景中的具体表现。

# 第二章 背景知识与关键技术

2.1 页缓存与页描述符

在 Linux 内核中，所有对文件的 buffered 读写都经过页缓存。页缓存位于虚拟文件系统层之下、具体文件系统之上，将磁盘上的文件数据缓存在内存中，以减少对存储设备的直接访问。

页缓存以页描述符（struct page）为基本管理单位。每个页描述符对应 4KB 物理内存，代表文件数据在内存中的一个缓存单元。struct page 有三个与本文直接相关的核心字段：index 记录该页在文件内的页索引——对于文件偏移 offset 处的数据，页索引的计算方式为 offset / 4096（文件偏移除以页大小向下取整）；mapping 指向该页所属 inode 的地址空间（address_space），将页与文件关联；flags 通过一组状态位标记该页的当前状况——PG_uptodate 表示页中数据与磁盘一致或更新，PG_dirty 表示数据已被修改但尚未写回，PG_locked 表示该页正在被某条 I/O 路径操作，其他路径不得同时访问。

当应用通过 read() 系统调用请求读取一段文件数据时，内核根据文件偏移计算出页索引，在 inode 的 address_space 中查找对应页描述符。如果存在且标记为 uptodate，数据直接从页缓存拷贝到用户缓冲区。如果不存在，则触发缺页流程：内核分配新的页描述符，对其加锁（设置 PG_locked），然后请求下层文件系统提供该页对应的磁盘数据位置。文件系统完成映射后，内核组织 I/O 请求从磁盘读取数据，数据到达后将页标记为 uptodate 并解锁。页缓存中的数据就绪后，系统调用将数据拷贝到用户缓冲区后返回。

上述流程中的全部操作——分配、加锁、标记状态、查找映射——均以单个 4KB 页描述符为粒度。一次 64KB 的读请求意味着十六次分配、十六次加锁、十六次映射查询。这正是第一章所述第一层问题的物理根源：页缓存的管理粒度决定了 I/O 路径上的元数据开销的放大倍数。

2.2 F2FS 块映射

页缓存知道需要读取哪个文件偏移范围的数据，但不知道这些数据在磁盘上的物理位置。这个信息由文件系统层提供。F2FS 使用基于节点（node）的索引结构来维护文件逻辑位置到物理存储位置的映射关系。

每个文件的 inode 块中存储了指向数据块的直接指针和指向间接节点块的指针。间接节点块再指向更下一层的数据块或节点块，形成一个多级索引树。F2FS 通过节点地址表（Node Address Table，NAT）管理所有节点块的物理位置——NAT 维护从节点 ID 到节点块物理地址的映射，使文件系统能够在需要时快速定位并读取节点块。

当页缓存请求文件系统为某个文件偏移范围提供映射时，F2FS 的块映射入口函数 f2fs_map_blocks 执行以下流程：将文件偏移转换为文件内部逻辑块号（偏移量除以块大小，F2FS 块大小通常为 4KB）；从 inode 开始沿节点树逐级查找，直到定位到该逻辑块对应的直接节点条目；从该条目中读取物理块号，并统计该条目之后连续的、物理地址也连续的块数量。映射结果以逻辑块号、物理块号、连续块长度和区间标志（普通映射 MAPPED、空洞 HOLE、新分配 NEW_ADDR 等）的形式返回给页缓存层。

在传统 4KB 路径中，一次 f2fs_map_blocks 调用通常返回与一个页描述符对应的一段连续映射。页缓存拿到物理块号后，将映射结果交给块层，组织实际的磁盘读取请求。

2.3 bio 的多页 I/O 提交

映射结果到手后，内核需要向存储设备发出实际的读取命令。在 Linux 中，所有块设备的 I/O 请求均以 bio（block I/O request）的形式组织和提交。

bio 的核心数据结构包括两部分。第一部分是 bi_iter，记录本次 I/O 请求的目标区域：bi_sector 表示目标设备上的起始扇区号（以 512 字节为单位），由物理块号乘以 8 换算（一个 4KB 块等于 8 个 512 字节扇区）；bi_size 表示本次 I/O 的总字节数。第二部分是 bi_io_vec，这是一个 bio_vec 数组。每个 bio_vec 描述一段物理连续的内存区间，包含起始页指针 bv_page、页内偏移 bv_offset 和区间长度 bv_len 三个字段。一个 bio 最多可包含 256 个 bio_vec（BIO_MAX_VECS）。

bio 的数据结构设计天然支持多页 I/O，这种能力体现在两个层面。首先，多个物理地址不连续的页可以通过不同的 bio_vec 被同一个 bio 引用，合并为一次 submit_bio 提交。其次，bio_vec 自支持 多页扩展 bvec 以来，其 bv_len 不再限于一个页大小——当一组物理连续页被加入同一个 bvec 时，bv_len 直接扩展为多页总长度，而非拆分为多个单页 bvec。bio_add_page 会检查新页的物理起始地址是否与已有 bvec 的末地址连续：若连续，则直接扩展该 bvec 的 bv_len；若不连续或 bio 已满，则新增一个 bio_vec。

这意味着，一个 64KB 的物理连续内存区间可以被一个 bio 中的少量 多页扩展 bvec 完整覆盖，bvec 数组的遍历开销相对于单页路径大幅降低。

bio 构建完成后通过 submit_bio 提交到块设备队列。块设备驱动通过 DMA 将数据从存储设备传输到 bio 所描述的内存区域。

I/O 的完成通知是通过回调机制异步传递的，理解这一点对后续全文至关重要。当块设备完成一次 DMA 传输后，它并不直接返回给调用者，而是在中断上下文（或软中断 bottom half）中触发 bio 上预先注册的完成回调函数 bi_end_io。这个回调在中断上下文中执行——它**不持有页缓存层的任何锁**，也不知道当初发起这个 bio 的上层代码当前处于什么状态。因此从上层视角看，bio 的完成是一个异步事件：提交 bio 的代码已经返回了，页缓存对象的状态更新要等到未来的某个时刻由完成回调来写入。

对于读操作，完成回调的职责是将 bio 覆盖范围内的每一个页描述符的 PG_locked 标志清除，设置 PG_uptodate（表示内存中的数据已与磁盘同步），然后解锁该页。从这一刻起，上层代码可以安全地读取这些页中的数据。对于写操作，完成回调的职责是清除页描述符上的 PG_writeback 标志——表示该页的脏数据已安全写入磁盘，写回流程可以宣告结束，该页后续可以重新被标记为脏进入下一轮写回，或在内存紧张时被回收。

这个异步完成机制有一个本文反复触及的关键后果：一个 bio 的完成回调只知道"我覆盖了哪些页"，但它不知道这些页是否属于一个更大的 folio，也不知道同一个 folio 是否还有其他 bio 尚未完成。在传统 4KB 路径中，一个页就是一个独立对象，一个 bio 覆盖一个页，一次回调即为一次完整的生命周期事件——这不构成问题。但当页缓存对象变成 Large Folio 后，一个 folio 的多个子范围可能被拆分到多个 bio 中，各 bio 的完成时机互不感知。于是，"整个 folio 的读是否完成"和"整个 folio 的写回是否结束"不再能从单个回调中判定——必须有一个跨回调的原子计数机制来汇总所有相关 bio 的完成状态。这正是第三章多处分析的问题起点，也是第四章统一状态结构中 read_bytes_pending 和 write_bytes_pending 计数器的设计动机。

2.5 走读例子：一次 64KB buffered read 的完整路径

现在用前四节的知识串联一个具体例子来展示完整的读路径。假设应用调用 read(fd, buf, 65536)，请求从文件偏移 0 开始读取 64KB 数据。

第一步：系统调用经 VFS 进入通用读路径 generic_file_read_iter，该函数循环处理 64KB 的读范围。对于偏移 0 处的数据，计算页索引为 0（0 / 4096 = 0），在文件 inode 的 address_space 中查找页索引为 0 的页描述符。假设页缓存为空（冷读），未命中。

第二步：页缓存分配一个新的页描述符（index = 0），对其加锁（PG_locked）。预读窗口检测到顺序访问模式后，将这个 4KB 的单页请求扩展为更大的预读范围，一次性分配 index = 0 到 index = 15 共十六个页描述符。每个页描述符均被加锁，准备接收数据。

第三步：页缓存调用 F2FS 的 f2fs_readahead 获取映射。F2FS 将文件偏移 0 转换为逻辑块号 0，沿节点树查找到对应的直接节点条目，得到物理块号（假设为 P）。继续检查后续块：逻辑块 1 映射到 P+1，逻辑块 2 映射到 P+2……假设在本次调用中 F2FS 返回了连续 16 个块的映射结果（P 到 P+15）。

第四步：块层接到映射结果后建立 bio。将物理块号 P 换算为起始扇区号 bi_sector = P * 8。然后逐页调用 bio_add_page：如果内核为这十六个页分配了物理连续内存，bio_add_page 会将其中物理连续的页合并到同一个 multi-page bvec 中——bv_len 被扩展为多页总长度，而非拆分为十六个单页 bvec。bio 的 bi_size 为 65536。submit_bio 将 bio 提交到块设备队列。

第五步：块设备驱动通过 DMA 将 64KB 数据从闪存读入 bio 所描述的内存区域。bio 完成后，完成回调函数被触发，遍历 bio 所覆盖的每个页描述符，清除其 PG_locked 并设置 PG_uptodate。十六个页全部就绪。

第六步：read() 系统调用将页缓存中的数据逐页拷贝到用户缓冲区 buf。所有数据拷贝完成后，系统调用返回 65536。

这个走读揭示了一条关键的矛盾线：块层的 bio 有能力将十六个页的数据打包为一次 I/O 提交——物理连续页还可以被合并到少量 multi-page bvec 中，进一步降低 bvec 数组的遍历开销——但页缓存层和文件系统映射层仍然围绕每个 4KB 页描述符独立操作：分配、加锁、映射查询各发生十六次。这是第一章所提矛盾的直观体现：块层的聚合能力被上层细粒度管理模型所制约。

2.6 Large Folio：可变阶数动态大页

上述走读暴露了 4KB 页描述符粒度的瓶颈。如果页缓存的管理粒度扩大，情况会如何变化？Large Folio 正是为回答这一问题而提出的。

folio 是对传统 struct page 的类型安全升级。在 folio 出现之前，内核通过复合页（compound page）机制将多个连续 4KB 页组织为一个整体，但接口层面始终以 struct page * 传参——调用者拿到的指针既可能指向复合页的头页，也可能指向某个尾页，这种语义模糊迫使大量代码反复调用 compound_head() 来确定指针归属，增加了复杂性和出错风险。folio 在类型层面提供了明确保证：一个 struct folio * 指针必然指向一个完整缓存对象的头部——要么是 order-0 的单页，要么是复合页的头页，绝不指向尾页。这一保证消除了 compound_head() 的语义模糊，使编译器可以进行类型检查。

Large Folio 就是阶数大于等于 1 的 folio。阶数是伙伴分配器中的概念：order = 0 表示 2⁰ = 1 个 4KB 页，order = 2 表示 2² = 4 个 4KB 页即 16KB，order = 4 表示 2⁴ = 16 个 4KB 页即 64KB。当文件系统通过 mapping_set_large_folios 声明自身支持更大粒度的缓存对象后，页缓存在检测到对文件的持续顺序访问时，会根据 readahead 窗口的大小和内存空闲状况动态选择 order，分配 order ≥ 1 的 folio——阶数不是预设的，而是在运行时决定的。

回到上一节的走读：在 Large Folio 路径下，64KB 的读请求不再触发十六次分配和十六次加锁，而是由页缓存分配一个 order = 4 的 64KB folio——一次分配、一次加锁。由于 folio 对应的 64KB 物理内存是连续分配的，bio 构建时可以将整个 folio 区间合并到一个 multi-page bvec 中——bv_len = 65536，bv_page 指向 folio 的首个 struct page。如果 F2FS 能够返回覆盖整个 folio 范围的连续映射，那么块层的 I/O 组装也只需围绕这个 folio 进行一次映射查询和一次区间提交——这正是 Large Folio 回应“4KB 页缓存粒度过小”问题的核心路径。

## 2.4 闪存逻辑地址空间与 FTL

bio 提交到块设备队列之后，最终到达的是闪存存储设备。然而，块设备驱动程序所看到的连续逻辑地址空间，与闪存芯片内部的实际物理布局之间存在根本差异。这种差异由闪存转换层（Flash Translation Layer，FTL）在固件层面隐藏和管理。

NAND 闪存具有一个核心的不对称性：写入以页为单位（通常 4KB 至 16KB），但擦除必须以块为单位（通常包含 64 至 256 个页，即数百 KB 到数 MB）。一个闪存页在被擦除之前只能写入一次。若要原地修改一个页中的已有数据，必须先擦除整个包含它的擦除块——而擦除操作的延迟远高于写入（毫秒级对比百微秒级），且每个擦除块的可擦写次数有限。

FTL 的角色是向上层呈现一个普通的线性逻辑块设备，同时在内部维护从逻辑块地址（LBA）到物理闪存页的动态映射。当文件系统写入一个 LBA 时，FTL 将其映射到一个已擦除的空闲物理页，然后将原物理页标记为无效。FTL 在后台执行垃圾回收——将包含有效数据的页搬移到新的擦除块，然后擦除旧的擦除块以回收空间。整个过程对上层文件系统透明。

这种架构导致一个直接影响：如果上层文件系统频繁进行随机的小块原地覆盖写入，FTL 内部的垃圾回收压力会急剧增大，产生显著的写放大——实际写入闪存的数据量远超文件系统发出的写入量。相反，如果文件系统倾向于将写入组织为连续的大块追加，则 FTL 可以将这些写入映射到连续的物理区域，减少碎片和 GC 开销。这一认识直接导向下一节所讨论的 F2FS 设计选择。

## 2.5 走读例子：一次 64KB buffered read 的完整路径

现在用前四节的知识串联一个具体例子来展示完整的读路径。假设应用调用 `read(fd, buf, 65536)` 即请求从文件描述符 fd 对应的文件的偏移 0 开始读取 64KB 数据。

第一步：系统调用经 VFS 进入通用读路径 `generic_file_read_iter`，该函数循环处理 64KB 的读范围。对于偏移 0 处的数据，计算页索引为 0（0 / 4096 = 0），在文件 inode 的 address_space 中查找页索引为 0 的页描述符。假设页缓存为空（冷读），未命中。

第二步：页缓存分配一个新的页描述符（index=0），对其加锁（PG_locked）。然后触发预读（readahead）——预读窗口检测到顺序访问模式后，可能将这个 4KB 的单页请求扩展为更大的预读范围，一次性分配 index=0 到 index=15 共十六个页描述符。每个页描述符都被加锁，准备接收数据。

第三步：页缓存调用 F2FS 的 `f2fs_readahead` 获取映射。F2FS 内部将文件偏移 0 转换为逻辑块号 0，沿着节点树查找到对应的直接节点条目，得到物理块号（假设为 P）。继续检查后续块：逻辑块 1 映射到 P+1，逻辑块 2 映射到 P+2……假设在本次调用中 F2FS 返回了连续 16 个块的映射结果（P 到 P+15）。

第四步：块层接到映射结果后建立 bio。将物理块号 P 换算为起始扇区号 `bi_sector = P * 8`。然后逐页调用 `bio_add_page`：对于 index=0 的页，加入其对应的物理内存段；对于 index=1 的页，因其物理地址与 index=0 的页连续（内核分配物理连续页面），被合并到同一个 `bio_vec` 中。十六个页全部加入后，bio 的 `bi_size` 为 65536。`submit_bio` 将 bio 提交到块设备队列。

第五步：块设备驱动通过 DMA 将 64KB 数据从闪存读入 bio 描述的内存区域。bio 完成后，完成回调函数被触发，遍历 bio 所覆盖的每一个页描述符，将其 PG_locked 清除并设置 PG_uptodate，表示数据已就位。十六个页全部就绪。

第六步：`read()` 系统调用将页缓存中的数据逐页拷贝到用户缓冲区 buf。所有数据拷贝完成后，系统调用返回 65536。

从这个走读中可以清楚地看到：尽管块层有能力将十六个页的数据打包为一个 bio 提交，页缓存层和文件系统映射层却仍然围绕每个 4KB 页描述符独立操作——分配、加锁、映射查询各发生十六次。这是论文背景中反复提及的矛盾的直观体现。

## 2.6 Large Folio：可变阶数动态大页

上述走读暴露了 4KB 页描述符粒度的瓶颈。如果页缓存的管理粒度扩大，情况会如何变化？Large Folio 正是为回答这一问题而提出的。

folio 是对传统 struct page 的类型安全升级。在 folio 出现之前，内核通过复合页（compound page）机制将多个连续 4KB 页组织为一个整体，但接口层面始终以 struct page * 传参——调用者拿到的指针既可能指向复合页的头页，也可能指向某个尾页，这种语义模糊迫使大量代码反复调用 compound_head() 来确定指针归属，增加了复杂性和出错风险。folio 在类型层面提供了明确保证：一个 struct folio * 指针必然指向一个完整缓存对象的头部——要么是 order-0 的单页，要么是复合页的头页，绝不指向尾页。这一保证消除了 compound_head() 的语义模糊，使编译器可以进行类型检查。

Large Folio 就是阶数大于等于 1 的 folio。阶数是伙伴分配器中的概念：order = 0 表示 2⁰ = 1 个 4KB 页，order = 2 表示 2² = 4 个 4KB 页即 16KB，order = 4 表示 2⁴ = 16 个 4KB 页即 64KB。当文件系统通过 mapping_set_large_folios 声明自身支持更大粒度的缓存对象后，页缓存在检测到对文件的持续顺序访问时，会根据 readahead 窗口的大小和内存空闲状况动态选择 order，分配 order ≥ 1 的 folio——阶数不是预设的，而是在运行时决定的。

回到上一节的走读：在 Large Folio 路径下，64KB 的读请求不再触发十六次分配和十六次加锁，而是由页缓存分配一个 order = 4 的 64KB folio——一次分配、一次加锁。由于 folio 对应的 64KB 物理内存是连续分配的，bio 构建时可以将整个 folio 区间合并到一个 multi-page bvec 中——bv_len = 65536，bv_page 指向 folio 的首个 struct page。如果 F2FS 能够返回覆盖整个 folio 范围的连续映射，那么块层的 I/O 组装也只需围绕这个 folio 进行一次映射查询和一次区间提交——这正是 Large Folio 回应“4KB 页缓存粒度过小”问题的核心路径。

## 2.7 F2FS 的异地更新、GC 与压缩文件

2.4 节已说明闪存不能原地覆盖写入的根本原因：写入粒度与擦除粒度不对称，原地覆盖会触发 FTL 内部的读改写放大。F2FS 的设计直接回应了这一物理约束。

F2FS 采用日志结构文件系统的核心策略：所有数据写入都以追加方式写入当前活动日志段（active log segment）的新位置，不覆盖旧数据。当已有的文件数据被修改时，新数据写入新的物理块，旧块随后被标记为无效。这种策略确保文件系统层发出的写入始终是顺序追加的，减少了 FTL 层面的碎片化和写放大。节点地址表（NAT）负责维护文件内部逻辑块号到当前有效物理块号的映射关系——这正是 F2FS 实现异地更新的元数据基础。段信息表（SIT）记录每个段的块有效性和利用率，段摘要区（SSA）保存每个块的归属信息（属于哪个 inode 的哪个逻辑块），检查点（Checkpoint）则定期记录文件系统的一致状态以便崩溃恢复。

异地更新自然带来一个后果：随着时间推移，旧块不断变为无效，存储段中有效块和无效块混杂，需要不定期的空间回收。F2FS 的垃圾回收（Garbage Collection，GC）机制负责这一任务。GC 的基本流程是：通过 SIT 选择一个有效块比例最低的 victim 段，读取该段中仍然有效的块的数据，将其重新写入当前的活跃日志段，然后更新相应的 NAT 条目指向新位置，最后擦除 victim 段使其可重新用于分配。GC 不是一条独立的路径，它在搬移有效块时需要重新进入页缓存和文件系统映射流程——读有效块需要查询映射并构建 bio，写新位置需要更新元数据。

F2FS 还支持文件数据压缩。压缩以簇（cluster）为单位，一个簇通常包含 2 至 4 个连续的文件系统块。写入时，一个簇内的原始数据被压缩后以更小的体积写入磁盘；读取时，需要先将压缩数据段读入临时缓冲区，解压后再填充到目标页缓存对象中。簇的边界以文件系统块为单位定义，与页缓存对象的大小之间没有固定的倍数关系。

上述三种机制——异地更新、GC 和压缩文件——共同决定了 F2FS 不是一条简单的"读块→写块"直线，而是普通读写、后台搬移和压缩数据处理三个路径相互交叉的网络。这为后文讨论 Large Folio 适配的复杂性提供了必要的背景。

## 2.8 iomap：字节区间映射与 per-folio 状态追踪

在传统 Linux 文件系统实现中，buffer_head 被用来描述页内文件系统块与底层设备块的对应关系——每个文件系统块挂接一个 buffer_head 对象，记录其物理地址和状态。当一个页描述符对应一个块时，这种逐块组织方式足够简单；但当页缓存对象变成 64KB 的大粒度 folio 后，一个 folio 内部可能挂接十六个 buffer_head，遍历和维护的开销随对象变大反向恶化。

iomap 框架通过一个根本性的思路转变来解决这一问题：将 I/O 映射的语义从“逐块”提升为“逐区间”。

iomap 的核心数据结构是 struct iomap，它描述文件的一段连续区间从文件字节偏移到设备字节地址的映射关系。一个 iomap 包含：文件内的起始字节偏移 offset、区间长度 length、对应设备的字节地址 addr、块设备指针 bdev 以及区间类型 type——如 IOMAP_MAPPED（已有映射）、IOMAP_HOLE（空洞）、IOMAP_UNWRITTEN（预分配但未写入）等。区间的长度以字节为单位描述，不需要是块大小的整数倍。

iomap 通过迭代器模式驱动 I/O 路径。文件系统实现一组回调函数（iomap_ops）：iomap_begin 在给定文件偏移和长度下填充 iomap 描述符，返回一段连续映射；iomap_end 在 I/O 完成后执行收尾处理。核心迭代入口 iomap_iter 循环调用 iomap_begin，每次获得一个区间映射后交由上层消费（读取、写入或写回），消费完当前区间后自动推进偏移，继续请求下一段映射，直到整个 I/O 请求范围被覆盖完毕。

iomap 框架向外提供了三个主要入口函数，分别对应文件系统的三条核心 I/O 路径。iomap_readahead 服务于缓冲读路径——当页缓存检测到顺序访问模式并触发预读时，该函数通过 iomap_iter 获取映射区间，分配 folio 并提交磁盘读取，bio 完成后由 iomap 内部的 per-folio 状态对象逐块标记 uptodate 位图。iomap_file_buffered_write 服务于缓冲写路径——应用数据先拷贝到页缓存中的 folio 页面，该函数通过 iomap_iter 为写入范围获取映射区间，处理空间分配和脏区标记，写入完成后由 per-folio 的 dirty 位图记录哪些块已被修改。iomap_writepages 服务于脏页写回路径——当内核决定将脏页写入磁盘时，该函数遍历脏 folio，通过 iomap_iter 为每个脏块范围查询映射，组织写 bio 并提交。三个入口共享同一套 iomap_ops——文件系统只需实现一次 begin/end 回调逻辑，iomap 框架自动将其适配到读、写、写回三条路径的不同语义中。

iomap 框架为每个 folio 维护一个 per-folio 状态对象 iomap_folio_state，存储在 folio->private 中。该对象内部维护以文件系统块为粒度的 uptodate 位图。当一个 64KB 的 folio 包含 16 个 4KB 块时，位图允许部分块被标记为 uptodate、部分尚未填充。这意味着，如果某次读取只覆盖了 folio 的部分块，iomap 可以仅标记已读入的块为 uptodate，而非丢弃整个 folio——这正是对 2.7 节所述“压缩文件簇与页缓存对象之间没有固定倍数关系”这一复杂性的直接回应。

## 2.9 本章小结

本章从两个层面建立了后续讨论所需的全部背景知识。上半部分（2.1 至 2.5）围绕一次 64KB buffered read 的完整路径，依次讲解了页缓存与页描述符、F2FS 块映射、bio 多页 I/O 机制和闪存逻辑地址空间四个概念层，并用一个走读例子将它们串联，直观展示了"上层粒度过小抵消下层大粒度 I/O 能力"的矛盾。下半部分（2.6 至 2.8）依次引入了本文的三层技术基础：Large Folio 通过可变阶数扩大页缓存对象粒度，回应第一层问题；iomap 通过字节区间映射将文件系统的 I/O 组织从逐块模式提升为区间模式，回应第二层问题；F2FS 的异地更新、GC 和压缩机制则揭示了 iomap 通用框架与 F2FS 私有语义之间的兼容挑战，这正是第三章所要分析的第三层问题。
